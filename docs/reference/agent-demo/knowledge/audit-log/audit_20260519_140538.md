## 1. セキュリティの脆弱性

### 【重要度: 中】localStorage データの無検証読み込み

**ファイル**: `src/hooks/useTodos.js` — `loadState()` 関数（44〜51行目）

```js
function loadState() {
  try {
    const raw = localStorage.getItem(STORAGE_KEY);
    return raw ? JSON.parse(raw) : initialState;
  } catch {
    return initialState;
  }
}
```

`JSON.parse` の成功のみを検証しており、返されたオブジェクトが `AppState` の形状（`todos: Todo[]`, `filter: FilterType`）に適合するかを検証していない。

- `todos` が配列でない場合、後続の `state.todos.map()` や `state.todos.filter()` でランタイムエラーが発生する。
- `filter` に `"all" | "active" | "completed"` 以外の値が入ると、フィルタリングロジックが意図しない挙動を示す。
- ブラウザ拡張機能や DevTools による localStorage 改ざんで、アプリがクラッシュする可能性がある。

**推奨対策**: `loadState` でスキーマ検証を実施し、不正なデータは `initialState` にフォールバックする。

### 【重要度: 低】XSS リスク（現状は安全だが注意点あり）

**ファイル**: `src/components/TodoItem/TodoItem.jsx`

```jsx
<span className={styles.text}>{todo.text}</span>
<button aria-label={`${todo.text} を削除`}>
```

React の JSX はテキストコンテンツと属性値を自動エスケープするため、現時点で XSS の危険はない。`dangerouslySetInnerHTML` も使用されていない。ただし、将来的に Markdown レンダリングや HTML 挿入を追加する場合はサニタイズが必須となる。

## 2. React ベストプラクティス違反

### 【重要度: 中】エラーバウンダリなし

**ファイル**: `src/App.jsx`

アプリケーション全体にエラーバウンダリが設置されていない。`useTodos` フックや各コンポーネントが予期せぬエラーをスローした場合（例: localStorage の不正データによるランタイムエラー）、React ツリー全体がアンマウントされ、ユーザーには白画面のみが表示される。

**推奨対策**: `App.jsx` を `ErrorBoundary` コンポーネントでラップし、最低限のフォールバック UI を提供する。

### 【重要度: 低】TodoContext で変動する state と安定した参照を混在

**ファイル**: `src/context/TodoContext.jsx`, `src/App.jsx`（20行目）

```jsx
<TodoContext.Provider value={{ filter: state.filter, dispatch }}>
```

`filter`（レンダリングごとに変わりうる値）と `dispatch`（`useReducer` が保証する安定した参照）を同一 Context に混在させている。`filter` が変化すると、`dispatch` のみを使用する `TodoForm` も不要な再レンダリングを起こす。

**推奨対策**: `TodoDispatchContext` と `TodoFilterContext` に分割する。

### 【重要度: 低】インライン無名関数のレンダリング毎生成

**ファイル**: `src/components/TodoItem/TodoItem.jsx`（13, 18行目）、`src/components/TodoFilter/TodoFilter.jsx`（19行目）

```jsx
onChange={() => dispatch({ type: "TOGGLE_TODO", id: todo.id })}
onClick={() => dispatch({ type: "DELETE_TODO", id: todo.id })}
onClick={() => dispatch({ type: "SET_FILTER", filter: value })}
```

各レンダリングで新規の関数インスタンスが生成される。現在の規模では体感的な問題は起きないが、リストが大規模になる場合は `useCallback` の使用を検討する。

### 【重要度: 低】`aria-current` に boolean ではなく文字列 `"true"` を使用

**ファイル**: `src/components/TodoFilter/TodoFilter.jsx`（22行目）

```jsx
aria-current={filter === value ? "true" : undefined}
```

WAI-ARIA の仕様上 `"true"` は有効なトークン値であり、React が DOM へ `aria-current="true"` と出力するため機能的に正しい。しかし React 慣例では boolean props に `{true}` を使うことが推奨される。

## 3. パフォーマンス上の問題

### 【重要度: 低】`filter` 変化時にも localStorage 全体を書き込む

**ファイル**: `src/hooks/useTodos.js`（56〜58行目）

```js
useEffect(() => {
  localStorage.setItem(STORAGE_KEY, JSON.stringify(state));
}, [state]);
```

`state` オブジェクト全体を依存配列に指定しているため、`filter` が変化しただけでも localStorage への書き込みが発生する。`filter` の永続化が不要であれば `todos` のみを保存する方が効率的。現規模では問題にならないが、タスク数が増えると書き込みコストが増大する。

### 【重要度: 低】`text.trim()` の毎レンダリング二重評価

**ファイル**: `src/components/TodoForm/TodoForm.jsx`（12, 27行目）

```jsx
const trimmed = text.trim();   // handleSubmit 内
disabled={!text.trim()}        // JSX 内
```

`text.trim()` が 1 レンダリングあたり 2 回実行される。`const trimmed = text.trim()` を一度だけ宣言してコンポーネントスコープで使い回す、または `useMemo` / `useState` で管理することで回避できる。

## 4. 要件との乖離

### 【重要度: 高】TypeScript 未使用（設計書との重大な乖離）

**設計書**: `knowledge/design/design.md` — セクション 4「ファイル構成」

設計書は全ファイルを TypeScript (`.tsx` / `.ts`) で実装することを明示している。しかし実装はすべて JavaScript (`.jsx` / `.js`) で行われており、型安全性が担保されていない。

- 型定義ファイル `src/types/todo.ts` が存在しない
- `Todo`, `FilterType`, `Action`, `AppState` の型定義がなく、prop の型誤りをコンパイル時に検出できない
- 設計書に記載された TypeScript 型定義が実装に活用されていない

**影響**: 設計書が定義した型安全性の恩恵が一切得られず、バグ混入リスクが高まる。

### 【重要度: 高】`types/` ディレクトリ・型定義ファイルが未実装

**設計書**: `knowledge/design/design.md` — セクション 2「データ構造」

設計書は `src/types/todo.ts` に `Todo`, `FilterType`, `Action` 型を定義することを規定している。このファイルが存在しないため、設計書の型定義がコードに反映されていない。

### 【重要度: 中】`createdAt` フィールドがソートに使用されていない

**設計書**: `knowledge/design/design.md` — セクション 2「データ構造」

設計書は `createdAt: number` を「ソート用」と明記している。しかし実装の `ADD_TODO` アクションは新 Todo を配列の先頭に追加するだけで（`useTodos.js` 14〜23行目）、`createdAt` を使った明示的なソートは行われていない。

- `createdAt` フィールドは `ADD_TODO` で生成されるが（`useTodos.js` 20行目）、フィルタリング・ソート・表示のいずれにも使用されていない。
- 「順序を明示的に管理する」という設計意図が実現されていない。

### 【重要度: 低】`main.tsx` / `src/index.css` が `src/` 内に存在しない

**設計書**: `knowledge/design/design.md` — セクション 4「ファイル構成」

設計書には `main.tsx`（エントリポイント）と `src/index.css`（グローバルリセット）が含まれている。`src/` ディレクトリ内に該当ファイルが確認できず、グローバルスタイルリセットが適用されているか不明。

## 総評

セキュリティ面では React の標準的な XSS 防御機構が機能しており、`dangerouslySetInnerHTML` の不使用は適切。ただし localStorage からの読み込みデータに対するスキーマ検証が欠如しており、アプリの堅牢性に懸念がある。

最大の問題は **TypeScript 未使用** と **型定義ファイルの未実装** であり、設計書が明確に指定した型安全性の仕組みが完全に欠落している。これにより設計書の意図が実装に反映されておらず、バグの混入リスクが高い状態にある。

`createdAt` フィールドが生成されながら未使用である点も、設計書との乖離として修正が必要。
