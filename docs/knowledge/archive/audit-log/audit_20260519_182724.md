## 指摘一覧

### [HIGH-001] 発言エンドポイントの認可バイパス（app/api/cases/[id]/argument/route.ts:5-27）

- **内容**: `POST /api/cases/[id]/argument` はセッション検証をいっさい行わず、`body.role`（"plaintiff" / "defendant"）をクライアントの自己申告値のまま受け入れる。現在のターンと一致するロールを申告すれば、当該ケースと無関係の第三者・匿名ユーザーが原告・被告のどちらにでも成りすまして発言を投稿できる。`createAdminClient()`（5行目）により RLS も無効化されており、DB 側のアクセス制御も機能していない。非機能要件「Row Level Security（RLS）を全テーブルで有効化する」に直接抵触する。
- **影響**: ケース ID（URL）を知っている攻撃者は、ゲーム中の任意のタイミングで任意の発言内容を注入できる。話し合いの完全性・公正性が根本から失われ、AI 判決の信頼性も消滅する。エンドユーザーは自分の話し合いが改ざんされていることに気づけない。
- **修正案**: セッションクライアントで `getUser()` を呼び、`user.id` と `c.plaintiff_id` / `c.defendant_id` を照合する。ゲスト参加者については、参加時（PATCH エンドポイント）にサーバー発行の署名付きトークンをセッションストレージ等に渡し、引数送信時にそのトークンを検証する方式に変更する。

---

### [MEDIUM-001] validateApiKey が全例外を「無効キー」と判定（lib/claude.ts:4-16）

- **内容**: `catch` ブロック（13行目）がすべての例外型を捕捉して `false` を返す。Anthropic API がネットワーク障害・レートリミット（429）・一時的サーバーエラー（500）を返した場合も `AuthenticationError` と同等に扱われる。
- **影響**: 有効な API キーを持つユーザーが Anthropic API の一時障害時にキーを保存できなくなる。「API キーが無効です」というエラーメッセージが表示され、ユーザーがキーを誤って無効と判断して再発行する恐れがある。
- **修正案**: `instanceof Anthropic.AuthenticationError`（または HTTP ステータス 401）のみ `false` を返し、それ以外は例外を再スローするか `{ valid: boolean; transient: boolean }` 型で返す。呼び出し元でも `transient === true` の場合は「一時的なエラーです。しばらくしてから再試行してください」と案内する。

### [MEDIUM-002] PATCH /api/profile の displayName に長さ検証なし（app/api/profile/route.ts:11）

- **内容**: リクエストボディから取り出した `displayName` に対して文字数・形式の検証がなく、任意長の文字列が DB に書き込まれる。`apiKey` には検証（`validateApiKey`）が追加されたが、`displayName` は取り残されている。
- **影響**: 認証済みユーザーが意図的に巨大な文字列を送信した場合、DB 書き込み負荷・ネットワーク転送量の増大が生じる。また、ヘッダー・チャット画面で `truncate` 処理がない箇所では UI 崩れが発生する。
- **修正案**: `displayName.trim().length > 50`（表示名の実用上の上限）で 400 を返すバリデーションを追加する。空文字・空白のみも `trim() === ""` でガードする。

### [MEDIUM-003] verdicts テーブルへの UNIQUE 制約が未適用（supabase/）

- **内容**: 設計書「データモデル変更」節に `ALTER TABLE verdicts ADD CONSTRAINT verdicts_case_id_unique UNIQUE (case_id)` が必要と明記されているが、現時点で未適用である。`app/api/cases/[id]/verdict/route.ts:16-18` の `phase !== "judging"` チェックは競合ウィンドウが存在し、2 秒ポーリングで同時に `judging` フェーズを検知した複数クライアントが並行リクエストを送信した場合、双方が判定チェックを通過してから phase を更新するより前に insert が実行されうる。
- **影響**: 1 ケースに 2 件以上の判決レコードが生成され、判決画面が不定挙動を起こす。勝者・スコアが異なる複数の判決が DB に残ることで、再表示時に表示内容が変わりうる。
- **修正案**: 設計書記載のマイグレーションを Supabase ダッシュボードで即時適用する。この作業はビルドの権限外のためリードからダイチへのエスカレーションが必要。constraint 適用後は、重複挿入時に DB 例外が発生するため verdict ルートで `catch` してから `409` を返す処理も追加すること。

### [LOW-001] GET /api/cases/[id] が内部 DB カラムをスプレッドで公開（app/api/cases/[id]/route.ts:35-43）

- **内容**: `buildCaseResponse` は `...c`（cases テーブルの全カラム）をレスポンスにスプレッドしている。`plaintiff_id`・`defendant_id`・`max_rounds`・`updated_at` などの内部フィールドがそのまま認証不要の GET エンドポイント経由で公開される。
- **影響**: ユーザー UUID の漏洩（クローラーによる収集の足がかり）。将来 DB カラムが追加された際に意図せず機密情報が公開されるリスクがある。
- **修正案**: レスポンスオブジェクトを明示的に構築し、必要なフィールド（`id`, `topic`, `phase`, `currentTurn`, `round`, `maxRounds`, `defendantId`, `plaintiff`, `defendant`, `arguments`, `verdict`, `createdAt`）のみを列挙する。

### [LOW-002] react-hooks/set-state-in-effect の lint 抑制（app/case/[id]/page.tsx:47,82,90）

- **内容**: `// eslint-disable-next-line react-hooks/set-state-in-effect` が 3 箇所に追加された。このルールは副作用内での `setState` が引き起こす無限レンダリングを検出するためのものであり、抑制することで将来の同種バグをサイレントに見逃す。
- **影響**: 現時点では無限ループは発生していないが、今後ポーリングロジックやロール復元ロジックを改修した際に無限レンダリングを誘発するリスクがある。
- **修正案**: 各 `useEffect` の依存配列と更新タイミングを見直し、lint ルールに準拠した実装（副作用外で状態を計算してから更新、または `useReducer` への切り替え）に変更する。
