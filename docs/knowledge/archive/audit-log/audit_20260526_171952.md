### [HIGH-001] FIX-1 の実装場所が task.md 指定と異なり、招待の発見が不可能（app/laws/page.tsx）

- **内容**:
  task.md（最優先）は FIX-1 を「`app/laws/page.tsx` に pending 招待セクションを追加する」と明示しているが、実装は `app/laws/[id]/page.tsx:38–68` にのみ `InvitationAccept` コンポーネントを追加している。`app/laws/page.tsx`（全109行）には `law_invitations` への参照が一切なく、招待の取得・表示コードが存在しない。

  ユーザーへの影響：
  - 招待されたユーザー B が `/laws` を訪れても pending 招待は表示されない。
  - B が招待を確認・承認するには `/laws/{lawId}` を直接知る必要がある。メール通知はスコープ外のため、A がサイドチャネル（口頭・別ツール）で URL を共有しない限り B は招待を発見できない。
  - 招待承認フローは通常のアプリナビゲーションから到達不能であり、L-2（メンバー招待）機能が実質的に不完全なままリリースされる。

  テスト偽陽性の補足：
  - `tests/e2e/laws.spec.ts:72–76`（L02）および `tests/e2e/laws.spec.ts:189–193`（L04）は承認ボタン存在チェックに `if (await acceptBtn.isVisible({ timeout: 5_000 }))` という条件分岐を使用している。ボタンが存在しなくても処理はスキップされてテストが継続する。
  - `tests/e2e/laws.spec.ts:81` の `await expect(memberCount).toBeVisible()` は `text=/メンバー\s+\d+人/` にマッチする「メンバー 1人」（User A 単独）でも真となるため、User B の参加未確認のままテストが通過する。
  - 結果として test-to-aud.md が報告する「CRITICAL-L02 ✅ 通過」は機能の正常動作を保証していない。

- **修正案**:
  `app/laws/page.tsx` の Server Component 部分に以下を追加する。

  1. `law_invitations` から `invitee_id = user.id AND status = 'pending'` のレコードを取得し、関連する `laws.name` および `laws.owner_id → profiles.display_name` を JOIN で取得する。
  2. 取得した招待を一覧表示するセクション（例：「届いた招待」）を法律一覧の上部に配置する。
  3. 承認・拒否ボタンは既存の `InvitationAccept` コンポーネントを再利用するか、同等の Client Component を `/laws/_components/PendingInvitations.tsx` として切り出す。
  4. `tests/e2e/laws.spec.ts:72–76` の `if (await acceptBtn.isVisible(...))` を `await expect(acceptBtn).toBeVisible()` に変更し、機能不全を検出できる hard assertion に修正する。

### [MEDIUM-001] E2E テストの critical シナリオで hard assertion が未使用（tests/e2e/laws.spec.ts）

- **内容**:
  L02・L03・L04 の各テストで、機能の成否を判定すべきアサーションが `if (await locator.isVisible(...))` による条件分岐で実装されており、対象 UI が存在しなくてもテストが「通過」する。

  | 行番号 | テスト | 問題の条件分岐 |
  |--------|--------|----------------|
  | 72–76  | L02    | `if (await acceptBtn.isVisible({ timeout: 5_000 }))` |
  | 119–122 | L03   | `if (await acceptBtn.isVisible({ timeout: 5_000 }))` |
  | 139–141 | L03   | `if (await voteBtn.isVisible({ timeout: 5_000 }))` |
  | 147–151 | L03   | `if (await article.isVisible({ timeout: 5_000 }))` |
  | 189–193 | L04   | `if (await acceptBtn.isVisible({ timeout: 5_000 }))` |

  HIGH-001 が示すように、L02・L04 の招待承認ボタンは `/laws` ページに存在しないため `isVisible` は常に false を返しているが、テストは全シナリオを通過し `test-to-aud.md` に「8/8 通過」として報告された。テスト結果が品質保証として機能していない。

- **修正案**:
  上記の `if` ブロックを `await expect(locator).toBeVisible({ timeout: 10_000 })` に置き換える。フロー上「存在しない場合もある」UI は別途コメントで意図を明示した上で conditional check の使用を局所化し、機能の成否を判定する箇所では必ず `expect` を使用する。

### [MEDIUM-002] PATCH /api/laws/[id]/invitations/[invId] で URL の lawId が検証に使用されていない（app/api/laws/[id]/invitations/[invId]/route.ts:10）

- **内容**:
  ルートハンドラーは `const { invId } = await params` と `invId` のみを展開し（行10）、URL 上の `[id]`（lawId）は一切参照していない。招待の検索クエリも `invId` 単独で行われる（行28–33）。

  これにより `PATCH /api/laws/<任意の有効UUID>/invitations/<実際の招待ID>` というリクエストが受理される。`law_members` への追加には `invitation.law_id`（DB 値）が使われるため操作結果は正しいが、URL パスと実際の操作対象法律が乖離する path confusion 状態にある。

  設計書は `[id]` と `[invId]` の対応を前提とした URI 設計を採用しており、他の招待関連ルート（`POST /api/laws/[id]/invitations/route.ts`）では `lawId` を一貫して使用しているため、この不整合は意図的な設計ではなく実装上の見落としと判断する。

- **修正案**:
  `app/api/laws/[id]/invitations/[invId]/route.ts:10` を `const { id: lawId, invId } = await params;` に変更し、招待の検索クエリに `.eq("law_id", lawId)` を追加して URL の law ID と招待の法律 ID が一致することをアプリ層で保証する。

  ```typescript
  const { data: invitation } = await admin
    .from("law_invitations")
    .select("id, law_id, invitee_id, status")
    .eq("id", invId)
    .eq("law_id", lawId)  // 追加
    .maybeSingle();
  ```

### [LOW-001] API ルートの URL パラメータに UUID バリデーション未実施（複数ファイル）

- **内容**:
  リクエストボディの `invitee_id`（`app/api/laws/[id]/invitations/route.ts:4`）および `new_owner_id`（`app/api/laws/[id]/owner/route.ts:4`）には UUID 形式の正規表現チェックが実装されているが、URL パラメータ `lawId`・`propId`・`invId` には同等のバリデーションがない。

  無効な UUID（例: `"abc"` や SQLインジェクション的な文字列）が URL に含まれた場合、Supabase が DB レベルで型エラー（`invalid input syntax for type uuid`）を返す。一部ルートはこのエラーを `data` の null チェックで 404 として処理するが、`error` フィールドを明示的にチェックするルート（`app/api/laws/[id]/route.ts:30–34`）では 500 レスポンスを返す可能性がある。エンドユーザーに対して誤ったエラーコードを返すことで実装の詳細が露出しうる。

- **修正案**:
  各ルートの冒頭で URL パラメータを UUID_REGEX で検証し、不正値には 400 を返す。あるいは共通バリデーションユーティリティ（例: `lib/utils/validate.ts`）を作成して全 API ルートで再利用する。

## 総評

FEAT-003 の基盤実装（認証・認可・RLS・合意チェック・退会処理の順序遵守）は設計書に沿って正確に実装されており、その点は評価できる。しかし FIX-1 が task.md の指示と異なる場所（`/laws/[id]/page.tsx`）に実装された結果、招待承認フローがユーザーから到達不能になるという重大な機能不全を抱えたままテストが通過している。テスト自体の assertion 設計に根本的な問題（`if` 条件分岐による偽陽性）があり、品質ゲートとして機能していない。HIGH-001 の解消とテストの hard assertion 化を最優先で対応すること。
