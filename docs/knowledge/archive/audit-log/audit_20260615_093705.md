## 指摘一覧

### [HIGH-001] `next` クエリパラメータに対する open redirect 脆弱性（app/auth/login/page.tsx:31-32）

- **内容**:
  `handleLogin` 内で以下のように `searchParams.get("next")` の値を一切検証せず `router.push(next)` に渡している。

  ```tsx
  const next = searchParams.get("next") || "/";
  router.push(next);
  ```

  task.md `## 監査観点` (3) と design.md `BUG-007 対応 → 監査観点` でも明示的にレビュー観点として挙げられている論点。引き継ぎメモ（eng-to-aud.md, test-to-aud.md）は「`router.push` は内部パス前提なので問題ない」「テストで相対パスは通過した」と結論付けているが、これは **誤り** で次の経路で実際に外部サイトへ遷移する。

  Next.js App Router の `useRouter().push()` は、引数 URL を `new URL(href, location)` 相当で解析し、**現在の origin と異なる場合は `window.location` 経由のフルナビゲーションに fallback する** 仕様。したがって以下が成立する。

  | 攻撃ペイロード | 解釈 | 結果 |
  |---|---|---|
  | `/auth/login?next=https://evil.example.com` | absolute URL（別 origin） | `https://evil.example.com` にフル遷移 |
  | `/auth/login?next=//evil.example.com/phish` | protocol-relative URL（別 origin） | `https://evil.example.com/phish` にフル遷移 |
  | `/auth/login?next=/\evil.example.com` | 一部ブラウザで `//evil.example.com` 同等に解釈 | 同上 |
  | `/auth/login?next=javascript:alert(document.cookie)` | `javascript:` URI | 一部ブラウザで `window.location` fallback 時に JS 実行（XSS） |

  攻撃シナリオ:
  1. 攻撃者は `https://<本物のドメイン>/auth/login?next=//evil.example.com/fake-login` を被害者に送る（ドメイン部分は正当なため信用される）。
  2. 被害者は正規のログインフォームで認証情報を入力しログイン成功する。
  3. ログイン直後に攻撃者ドメインへフル遷移し、攻撃者は「セッション切れです、もう一度ログインしてください」等の偽フォームで認証情報の再収集を行う。

  E2E spec `tests/e2e/auth-login.spec.ts:52-69`（BUG-007-2）は `?next=/history` の **正常系のみ** を検証しており、外部 URL のケースは未検証。test-to-aud.md L416-418 の「現時点で脆弱性は検出されず（リスク評価: 低）」「外部 URL の脆弱性は検出されず」は根拠なし（外部 URL のテストを実行していない）。

  **重大度判定の理由 (HIGH)**: CWE-601（URL Redirection to Untrusted Site）/ OWASP A03:2021 のうち、**ログイン直後** に発火する redirect は信頼境界が最も高い瞬間に攻撃者ドメインへ遷移させるため、典型的な MEDIUM の open redirect より phishing 成功率が高い。さらに `javascript:` 経由で XSS にもエスカレーション可能。task.md `### 監査観点` (3) でリード自身が「脆弱性ありと判断した場合はガード追加を推奨」と明記しているとおり、未実装の本ガードは必須。

- **修正案**:
  `router.push(next)` の前で `next` をホワイトリスト方式で検証する。最小修正は以下。

  ```tsx
  const rawNext = searchParams.get("next");
  // 同一 origin の内部パスのみ許可:
  //   - "/" で始まる
  //   - "//" や "/\" (protocol-relative の罠) で始まらない
  //   - 制御文字を含まない
  const isSafe =
    typeof rawNext === "string" &&
    rawNext.startsWith("/") &&
    !rawNext.startsWith("//") &&
    !rawNext.startsWith("/\\");
  const next = isSafe ? rawNext : "/";
  router.push(next);
  ```

  あわせて E2E spec に以下のリグレッションケースを追加することを推奨:
  - `?next=https://evil.example.com` → ログイン後に同 origin の `/` に着地すること
  - `?next=//evil.example.com` → 同上
  - `?next=javascript:alert(1)` → JS が実行されず `/` に着地すること

### [LOW-001] `useSearchParams()` を使う Client Component が Suspense 境界に包まれていない（app/auth/login/page.tsx:8-10）

- **内容**:
  `app/auth/login/page.tsx` は page 直下の Client Component で `useSearchParams()` を呼んでいる。Next 16（環境定義書 L7: 16.2.6）では `useSearchParams` は呼び出し元のサブツリーを **最寄りの `<Suspense>` 境界まで強制的にクライアント描画にダウングレード** する。本ページは page コンポーネント自体が `"use client"` のため、サブツリーの境界が存在せず、ページ全体が prerender からオプトアウトされる。

  既存 `app/case/[id]/CaseRoom.tsx` が同パターンで build を通過しているため build SUCCESS 自体は問題ないが、設計書 `## BUG-007 対応 → 監査観点` (2) が掲げた「build 警告が出ていないか」観点として、将来 Next が同パターンを **error 化** する可能性に備えた予防的指摘。test-to-aud.md L368 の「Suspense 警告なし」は dev server 実行時の確認のみで、production build のフル静的解析を通したログは引き継ぎに含まれない。

  **重大度 LOW の理由**: 現状の build / E2E は通過しており、ユーザーへの直接的な機能影響・セキュリティ影響はない。ただし将来 Next が仕様を厳格化した場合に他の `useSearchParams` 使用箇所と合わせて一括対応が必要になる予兆として記録する。

- **修正案**:
  本 PR のスコープを超えるため対応必須ではない。将来対応時は以下のいずれか。
  - フォーム部分を子コンポーネントに切り出し、page 側で `<Suspense fallback={...}><LoginForm /></Suspense>` で包む
  - `searchParams` を Server Component（page）で受け取り、props として Client Component に渡す（Next 推奨パターン）

## 総評

BUG-007 の修正方針自体は妥当で、`router.refresh()` 削除の根拠も筋が通っており、`next` 対応も前方互換構成として綺麗にまとまっている。

しかし task.md / design.md / handoff いずれもが明示的に挙げていた監査観点「open redirect」について、ガード未実装のまま「`router.push` は内部パス前提なので大丈夫」と結論付けて引き継がれている点は重大な誤認である。Next.js の `router.push` は別 origin URL を `window.location` フル遷移に fallback する仕様であり、ログイン直後のフル遷移は phishing 成立条件として極めて好都合な瞬間にあたる。HIGH-001 のガードを `next.startsWith("/") && !next.startsWith("//")` 等で必ず追加した上で、外部 URL・protocol-relative URL・`javascript:` URI に対する E2E リグレッションを spec に組み入れてから再監査されたい。

LOW-001 は将来の Next 仕様変更に備えた予防的記録で、本 PR の通過判定とは独立。

判定: **不通過**（HIGH 1 件）。HIGH-001 修正後の再監査を要する。
