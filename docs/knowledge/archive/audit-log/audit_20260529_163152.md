## 指摘一覧

### [LOW-001] `friends` 系ルートで UUID ガードが認証・ボディ検証の後段にあり、task.md の「各メソッドハンドラ先頭」要件を満たさない（`app/api/friends/requests/[id]/route.ts:9-31`、`app/api/friends/[id]/route.ts:11-16`）
- **内容**:
  - task.md L44 は「各 HTTP メソッドハンドラ先頭にガードを置く」「ゲスト経路・認証経路の双方で漏れなく適用」と指示し、eng-to-aud.md 判断 4 も「`await params` 直後・あらゆる DB アクセスより前」と標榜している。
  - laws 系・cases 系（例: `app/api/cases/[id]/route.ts:14-17`）は宣言どおりハンドラ最先頭で `isUuid` ガードを通すが、`friends` 系 2 ルートだけは構造が異なる。
    - `app/api/friends/requests/[id]/route.ts`: `auth.getUser()`（L9-11）→ ボディ解析（L13-18）→ `action` 検証（L20-26）→ **その後** L28-31 でようやく UUID ガード。
    - `app/api/friends/[id]/route.ts`: `auth.getUser()`（401 返却）→ その後に UUID ガード。
  - DB アクセス前ではあるため PostgreSQL エラー漏洩（LOW-001 由来の本質的懸念）は防げているが、**「不正 ID は一律 400」という契約が route family 間で不統一**になる。具体的には未認証＋不正 ID のリクエストで、laws/cases は 400 を返すのに対し `friends` 系は 401 を先に返す。テスタ引き継ぎ S2／S4 が掲げる「不正 ID は DB 到達前に一律 400」の前提とズレが生じる。
  - エンドユーザー影響は軽微（実害なし）だが、エラー契約の一貫性が崩れ、将来 `friends` ルートにガード前 DB アクセスが追加された際にガード漏れを誘発しやすい配置である。
- **修正案**: `friends` 系 2 ルートでも `const { id } = await params` とその `isUuid` ガードを `auth.getUser()` より前（ハンドラ最先頭）へ移動し、laws/cases 系と配置・返却コード順序を揃える。UUID 形式自体は秘匿情報ではないため、認証前に 400 を返しても情報漏洩はない。

### [LOW-002] fetch 失敗時のエラーメッセージが「時間をおいて再度お試しください」固定で、403/404/409 など恒久失敗時にユーザーを無意味な再試行へ誘導する（`app/laws/_components/PendingInvitations.tsx:33-39`）
- **内容**:
  - `respond()` は `res.ok` が false の場合に一律「操作に失敗しました。時間をおいて再度お試しください。」を表示する（L34）。
  - しかし当 PATCH（`app/api/laws/[id]/invitations/[invId]/route.ts`）が返す失敗は 403（他人宛招待）・404（招待が存在しない）・409（処理済み招待）が中心で、いずれも**時間をおいても解消しない恒久エラー**である。「時間をおいて再度お試しください」は一時的障害を示唆する文言であり、ユーザーを永続的に失敗する再試行へ誘導する。
  - 設計（design.md「LOW-002 設計」）は失敗時にリフレッシュを抑止して招待行を残す方針を採るため、ユーザーは同じ行を再操作し続け、同じ恒久エラーを繰り返す。これは task.md L32 が解消を狙った「連打を誘発する」事象を文言面で部分的に再生産している。
  - なお `res.ok` 検査・`rose-600`・`role="alert"`・成功時のみ `router.refresh()`・`finally` での `processingId` 解除は task.md/design.md どおり正しく実装されている。本指摘はメッセージ文言の妥当性に限定する。
- **修正案**: 文言から再試行の示唆を外し、状態不整合を表す中立的メッセージへ変更する（例: 「この招待は処理できませんでした。画面を更新してください。」）。可能なら 409/404（招待が既に消滅・処理済み）の場合のみ `router.refresh()` を呼んで陳腐化した招待行を消し、403 等は行を残してエラー表示、という分岐にすると task の「連打防止」意図と整合する。
