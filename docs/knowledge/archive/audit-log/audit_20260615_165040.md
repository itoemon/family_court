## 指摘一覧

### [MEDIUM-001] BUG-005 必須シナリオ #2/#3 が E2E spec に存在せず未検証（tests/e2e/bug005-closing-trigger.spec.ts:133-168）
- **内容**:
  task.md L108-116 の「必須」テスト観点として、以下 2 件が指定されている。
    - #2: 3 ラウンド完了 → 両者 finish → `phase=judging`。closing greeting 2 行 (`arguments`) と AI 閉廷宣告 1 行 (`judge_messages`) が挿入され、`arguments.created_at` < `judge_messages.created_at` の順序であることを assert。
    - #3: 早期 end-proposal 両者合意 → `phase=judging`。同様に挿入と順序を assert。

  実際の `tests/e2e/bug005-closing-trigger.spec.ts` を確認したところ、
    - `BUG-005-1` (L74-127) は argument フェーズ中に `judge_messages.trigger_type='closing'` が 0 件であることのみを検証する **負例テスト** のみ。
    - `BUG-005-2` (L133-145) と `BUG-005-3` (L151-168) は本体が `expect(true).toBe(true);` のみのダミー。コメント上で「実装確認」と称しているが、実コードを inspect しているわけではなく、アサーションは常に成功する。

  すなわち task.md「必須」#2/#3 と「推奨」#4 (AI 生成失敗フォールバック)、#5 (`extension_voting` 中の polling) は **end-to-end で一切実行されていない**。テスタ引き継ぎ (test-to-aud.md L18) は「3/3 通過 ✅」「パイプライン承認可」と報告しており、実態より良い結果に見える内容になっている（BUG-005-2/3 が 0ms 完了している点が証跡）。

  実装コード (`lib/case-closing.ts`, `app/api/cases/[id]/end-proposal/route.ts:153-192`, `app/api/cases/[id]/extension-vote/route.ts:180-219`) の静的レビュー、および `grep -rn "trigger_type.*closing"` が `lib/case-closing.ts:55` の 1 箇所に収束していることは確認済みで、実装ロジック自体に欠陥は見つからない。しかし「closing greeting → AI 閉廷宣告の挿入順序」「AI 生成失敗時に phase=judging 遷移が維持される」「`extension_voting` 中に closing が増えない」という本変更の本質部分が動的に検証されていないため、リグレッション検知能力が著しく弱い。

- **修正案**:
  `tests/e2e/bug005-closing-trigger.spec.ts` に以下のシナリオを実 spec として追加する。
    1. 既存 `BUG-005-2`/`BUG-005-3` のダミーを削除し、task.md 「必須」#2 を実装: 原告 ↔ 被告 が 3 ラウンド消化 → 延長投票で両者 finish → `judge_messages` の `trigger_type='closing'` 件数 = 1、`arguments` の `phase='closing', is_greeting=true` 件数 = 2 を admin client 経由で assert。さらに `arguments.created_at` < `judge_messages.created_at` も assert（AI 生成完了まで polling 待機）。
    2. 「必須」#3: 1 ラウンド目で原告が end-proposal → 被告が同意 → `phase=judging`。上記 #1 と同じ assert を行う。
    3. 「推奨」#4: `profiles.api_key_encrypted` を NULL に更新した状態で #2 を実行し、closing greeting 2 行は挿入されつつ `judge_messages.trigger_type='closing'` が 0 件で留まることを assert（ヘルパー L24-29 の早期 return パスを保証）。

  既存の `BUG-005-1` (argument フェーズ中の closing 不在) は維持。`describe.skip` の暫定凍結は不可。

## 総評
- 実装本体（`lib/case-closing.ts` のテーブル境界保護、`argument/route.ts` の `extension_voting` 遷移時 `judge_messages` INSERT 完全排除、両呼び出し経路での `insertClosingGreetingsForCase` → `insertClosingJudgeMessage` の順序、失敗時にも `phase=judging` を維持するエラーハンドリング）はいずれも task.md / design.md / eng-to-aud.md の指示にほぼ完全準拠で実装されており、設計通り。`grep` 検証で `trigger_type='closing'` の INSERT が `lib/case-closing.ts:55` 1 箇所に収束していること、`lib/case-closing.ts` 内に `arguments` テーブル / `DEFAULT_CLOSING_GREETING` への参照がないこと（テーブル境界保護）、`lastSpeakerRole` の SELECT が `is_greeting=false` で closing greeting 自身を巻き込まないことも確認済みで、コード面の品質は高い。
