## 指摘一覧

### [LOW-001] `restoreRole` の race condition 軽減コメントが欠落（app/case/[id]/CaseRoom.tsx:67-79）
- **内容**: マウント時の `restoreRole` useEffect は `[caseId]` 依存で GET `/api/cases/${caseId}` を非同期取得し `setMyRole` / `setCaseData` を行う。本 PR で `handleJoinAsAccount` / `handleJoinAsGuest` 直後に `setMyRole("defendant")` + `setCaseData(data)` + `await fetchDefenseMessages()` の系列が走るようになったため、極端に遅い `restoreRole` の応答が join 後の state を上書きするケースが理屈上発生し得る。サーバ応答内容（参加後の callerRole）と join 直後の state が等しいため実害は出ないが、本 PR の意図（参加直後の見え方を厳密に整える）と読み手側で意図対応が取りづらい構造になっている。仕組み上のリグレッションを防ぐため、`restoreRole` 側に「join 後の state を上書きしないこと」を保証するガード（例: 既に `myRole` が確定済みなら `setMyRole` をスキップ）を入れるか、コメントで race の許容理由を明示しておくことを推奨する。
- **修正案**: `restoreRole` 内で `setMyRole(data.callerRole)` を呼ぶ前に「既に `myRole` が確定している場合はスキップする」分岐を入れる、もしくは「サーバが callerRole=defendant を返す前提で上書きしても等価」というコメントを `app/case/[id]/CaseRoom.tsx:73` 周辺に追記する。本 PR スコープ外として別 backlog 化も可。

## 総評

BUG-004 の修正方針は妥当。`handleJoinAsAccount` / `handleJoinAsGuest` 双方で参加 PATCH 成功後に `await fetchDefenseMessages()` を明示呼び出しする実装は、リード調査で特定された「参加前 401/403 で `showDefenseTab=false` に倒れ、再 fetch されない」根本原因に対する直接的な対処になっている。

監査観点 3 点について個別所感:

1. **race condition の有無**: `setMyRole("defendant")` / `setCaseData(data)` → `await fetchDefenseMessages()` の系列は await による直列化で `setShowDefenseTab(true)` の順序が確実に保たれる。React 18+ の automatic batching と await microtask 境界の組み合わせで、中間状態（join 完了 → defense tab 出現）に一瞬の flicker は生じうるが UX 上問題なし。

2. **disable コメント削除の妥当性**: 初回オーディの LOW-002 で「`fetchDefenseMessages` 側だけ削除して `fetchCase` 側と不一致」を指摘した結果、**両 useEffect 側とも disable コメントを残して一貫性を回復** する判断（`CaseRoom.tsx:106`, `:116`）となっており妥当。lint の挙動厳格化に対しても両者同時に挙動が変わるため整合性が崩れない。

3. **参加前 401/403 ログのノイズ**: task.md で「本 PR では既存挙動維持」と明記されており、スコープ外として扱うのが正しい。改善余地は backlog 項目化で別 PR 対応が妥当。

差分は `+35 / -29` に拡大したが、初回オーディ LOW 3 件 (try/catch 分離、コメント不整合、関数移動) と 2 巡目 LOW 2 件 (setLoading 漏れ + コメント微調整) の消化を含んでいる構成で、いずれも動作変更は LOW-001 (silent fail 化) のみ。テスタ側で CRITICAL 4/4 + BUG-004 3/3 が通過しており、リグレッションも検出されていない。

判定: **通過**（HIGH 0 件、総指摘 1 件）。LOW-001 は本 PR スコープ外として別 backlog 化推奨。
