## 根本原因分析

### 🔴 **CRITICAL: Middleware が `/case` へのゲストアクセスをブロック**

**ファイル**: `middleware.ts:32-37`

```typescript
const PROTECTED_PATH_PREFIXES = ["/history", "/profile", "/case"];
const isProtected =
  pathname === "/" ||
  PROTECTED_PATH_PREFIXES.some(p => pathname === p || pathname.startsWith(p + "/"));
if (!user && isProtected) {
  return NextResponse.redirect(new URL("/auth/login", request.url));
}
```

**問題**:
- `/case/:id` パスが `PROTECTED_PATH_PREFIXES` に含まれている
- 未認証ユーザーはすべて `/auth/login` へリダイレクト
- ゲスト被告が被告参加選択画面（`/case/[id]/page.tsx` の `joinMode === "choose"` 画面）に到達不可

### 設計との矛盾

| 項目 | 設計書 | 実装状況 |
|---|---|---|
| **ゲスト参加フロー** | ✅ 明記（M04） | ✅ API 実装済み（`generateGuestToken`） |
| **被告参加選択画面** | ✅ 期待（アカウント選択 / ゲスト選択） | ❌ **未認証で到達不可** |
| **middleware の保護** | 記載なし | ❌ `/case` を全保護 |

### M01・M02 が通過した理由

- **認証済みユーザー**のみ参加（middleware をスルー）
- ゲスト参加（M03・M04）は未認証が前提なので、middleware でブロック

## 影響範囲

- **M03 失敗**: 第三者割り込み拒否フローが機能していない（ゲスト参加選択画面に到達不可）
- **M04 失敗**: ゲスト被告フロー全体が機能していない（Cookie トークン発行まで到達不可）
- **既存機能への影響**: M01・M02（認証済み参加）は正常。未認証ユーザーへの影響のみ

---

## テスタの結論

### ❌ パイプライン差し戻し（FAIL）

**ビルド（実装）側で修正が必須**:

ゲスト参加フロー（M03・M04）が機能していない。middleware で `/case/:id` への未認証アクセスが全てブロックされているため。

**修正案（優先順）**:

1. **Option A**: middleware の `/case/:id` をプロテクト対象から除外
   - 簡易: PROTECTED_PATH_PREFIXES から `/case` を削除
   - セキュリティ考慮: `/case/:id` が全員アクセス可能になる

2. **Option B（推奨）**: ゲスト被告参加フロー専用ルート作成
   - `/case/:id/join` など、未認証アクセス許可のルートを分離
   - 既存の `/case/:id` は認証必須のまま維持
   - セキュリティと機能のバランスが取れる

3. **Option C**: middleware で被告参加選択画面の判定を追加
   - `!myRole && caseData.phase === "waiting"` かつ被告未参加のケースは許可
   - 複雑だが最も制限的

**推奨**: Option B（既存アーキ・セキュリティとのバランス最適）
