## CRITICAL シナリオ実行結果

### [CRITICAL-M01] 2ユーザー間の会話フロー（両者認証済み）

- **結果**: ✅ **通過**
- **実行時間**: 15.2 秒
- **確認項目**:
  - ユーザー A（原告）がケースを作成 → `/case/:id?role=plaintiff` へリダイレクト
  - ユーザー B（被告）が同じ URL を開く
  - B が「アカウントでログインして参加」を 2 ステップで実行 → 参加完了
  - 参加直後は A のターン（opening フェーズ）で、B は「さんの返答を待っています」が表示される
  - A が「原告の最初の発言」を送信 → DB に記録され、画面に表示
  - B がリロード後、自分のターンになったことを確認して「被告の返答」を送信
  - A がリロード後、B の返答を確認 → フロー完全成功

**詳細**: 両ユーザー認証済みでのターン交代会話フローが正常に機能。セッション・ロール判定・DB 反映・ポーリング更新すべて正常。

### [CRITICAL-M03] 第三者の割り込み拒否

- **結果**: ✅ **通過**
- **実行時間**: 6.9 秒
- **確認項目**:
  - A（原告）がケースを作成
  - ゲスト（未認証）が「ゲストとして参加」で被告参加（defendant_guest_name セット）
  - B（認証済みの第三者。A でも A と B の関係者でもない）が同じ URL を開く
  - B にはケース topic が表示される（ページが読み込まれている）が、発言フォーム（textarea）は表示されない
  - B は observer 扱いで書き込み不可

**詳細**: RLS / 認可チェック正常。認証済みユーザーでも protagonist / defendant 権限がなければ write フォームが非表示。データベース側でも write リクエストを拒否（API 403 推定）。第三者の割り込みを完全に防止。

---

### [CRITICAL-M04] ゲスト被告フロー

- **結果**: ✅ **通過**
- **実行時間**: 6.5 秒
- **確認項目**:
  - A（認証済み原告）がケースを作成
  - ゲストコンテキスト（未認証）が URL を開く → 参加オプション表示
  - ゲストが「ゲストとして参加」を選択 → 名前入力「ゲスト太郎」→ 送信
  - PATCH /api/cases/:id { asGuest: true, defendant_guest_name: "ゲスト太郎" } が実行される
  - レスポンスの Set-Cookie に guest_defendant_{id} トークンが付与される
  - ゲスト側で発言フォームが表示される（Cookie トークンが有効な証拠）
  - A が発言を送信
  - ゲストがリロード後、自分のターンになったことを確認して返答
  - A がリロード後、ゲストの返答を確認

**詳細**: HMAC Cookie トークン方式でゲスト被告の認可を実現。トークン発行・Cookie 保持・リロード後の復元・ターン制御すべて正常。未認証でもケース参加・発言が可能（ゲスト限定）で、セキュリティと利便性を両立。
