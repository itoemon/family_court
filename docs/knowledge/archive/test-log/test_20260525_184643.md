### [CRITICAL-M03] 第三者の割り込み拒否
- **結果: ❌ 失敗**
- **失敗原因**: 
  - ゲスト被告参加時に `PATCH /api/cases/[id]` が **500 エラー**で応答
  - サーバーログ: `guest_tokens INSERT failed: permission denied for table guest_tokens`
  - フロント: タイムアウト（「{opponentName}さんの返答を待っています」が表示されない）
- **RLS 権限エラー詳細**:
  - `guest_tokens` テーブルの RLS ポリシーが Service Role のみに限定されている
  - しかし API の `generateGuestToken()` 呼び出し時点で、認証済みロール（authenticated）で実行されている
  - Supabase では Service Role 以外からの INSERT が RLS で ブロックされている
- **ビルド実装上の問題**:
  - `app/api/cases/[id]/route.ts` の PATCH ハンドラで `generateGuestToken()` を呼んでいるが、
    Supabase Admin Client（Service Role 権限）を使用していない可能性
  - app-side の admin client の初期化・使用を確認が必要

### [CRITICAL-M04] ゲスト被告フロー
- **結果: ❌ 失敗**
- **失敗原因**: M03 と同一（ゲスト参加時の RLS 権限エラー）
- **詳細**: ゲスト（未ログイン）が参加する際の `PATCH /api/cases/[id] { asGuest: true }` で 500 エラー

## 技術詳細

### M03・M04 共通エラー

**サーバーログ**:
```
generateGuestToken failed: Error: guest_tokens INSERT failed: permission denied for table guest_tokens
    at generateGuestToken (lib\guest-token.ts:23:20)
    at async PATCH (app\api\cases\[id]\route.ts:152:13)
```

**HTTP レスポンス**:
- M03: `PATCH /api/cases/4256718d-33bc-4de5-a572-d9d4ccfd5b5b 500 in 624ms`
- M04: `PATCH /api/cases/373ddf04-234f-45b6-99cb-fe224405f8ee 500 in 299ms`

### 根本原因の推測

1. **DB 側は正常**: `guest_tokens` テーブルは CREATE TABLE まで成功している（CREATE は anon/authenticated でも許可可能）
2. **RLS ポリシー設定**: INSERT を Service Role のみに限定した設定は正しい（セキュリティ要件を満たす）
3. **API 実装問題**: `app/api/cases/[id]/route.ts` の PATCH ハンドラで、
   - Supabase admin client（Service Role 権限）を使わずに INSERT しようとしている
   - あるいは admin client の初期化に失敗している
   - あるいは `generateGuestToken()` 内で admin client を使用していない

### 設計書・要件との整合性

- **設計書要件**: 「トークン発行は Admin Client（Service Role）で実行」（task.md 記載）
- **実装状況**: 実装ノート(eng-to-aud.md)では対応済みと記載されているが、動作テストで失敗
- **判定**: **実装の問題**（設計要件が不正ではなく、実装でスルーされている）
