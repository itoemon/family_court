## 根本原因

### 実装漏れ: PATCH /api/cases/[id] レスポンスが callerRole を含まない

**設計書**（design.md）の指示:
- GET `/api/cases/[id]` のレスポンスに `callerRole` フィールドを追加する
- `callerRole` は「plaintiff」「defendant」「observer」のいずれか
- サーバー側で完全に決定される

**実装状況**:
- ✅ GET ハンドラ: `callerRole` を正しく追加（app/api/cases/[id]/route.ts:48）
- ❌ PATCH ハンドラ: `callerRole` が漏れている（同ファイル:77, 95）

**クライアント動作**:
- 被告が参加ボタンを押す → PATCH リクエスト
- PATCH レスポンスに `callerRole` がない → `setCaseData()` でも callerRole が undefined
- 条件 `!canSpeak && myRole && !["waiting", "judging", "verdict"].includes(caseData.phase)` が満たされない
- 「相手の返答を待っています」が表示されない
- テスト：タイムアウト（10秒待機中に要素が現れない）

## 差し戻し指摘

### 対象ファイル
`app/api/cases/[id]/route.ts`

### 修正方法

**77行目（アカウント参加）**:
```typescript
// 変更前
return NextResponse.json({ ...caseData, defendantName: profile?.display_name });

// 変更後
return NextResponse.json({ ...caseData, defendantName: profile?.display_name, callerRole: "defendant" });
```

**95行目（ゲスト参加）**:
```typescript
// 変更前
const guestResponse = NextResponse.json(await buildCaseResponse(admin, id));

// 変更後
const guestResponse = NextResponse.json({ ...(await buildCaseResponse(admin, id)), callerRole: "defendant" });
```

### 設計書との照合

design.md の要件「API 仕様（変更・追加するエンドポイントのリクエスト/レスポンス定義）」では、GET ハンドラの変更内容を詳述しているが、PATCH ハンドラについては触れていない。しかし、クライアント側でページリロード時に確実にロールを復元するために、PATCH レスポンスにも `callerRole` が必須。
