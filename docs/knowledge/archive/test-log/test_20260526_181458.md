## 不合格の主要原因

### テスト環境前提条件未充足: フレンド関係が設定されていない

**実装は完全だが、テスト前準備が不足している**

#### 根本原因
- ユーザー A がユーザー B を「招待」しようとして表示名で検索（`tests/e2e/laws.spec.ts:59`）
- `POST /api/laws/[id]/invitations` では招待前に `friend_requests` テーブルでフレンド確認（design.md より）
- テストアカウント間に `friend_requests(status='accepted')` レコードが存在しないため、招待が作成されない
- よって B は pending 招待を受け取らず、L02 で「承認」ボタンが見つからない

#### 実装検証結果（コード確認）
✅ **FIX-A は完全実装済み**：
- `app/laws/page.tsx` が `law_invitations` から pending 招待を取得（lines 32-70）
- `app/laws/_components/PendingInvitations.tsx` は正しくレンダリング（line 113）
- コンポーネント実装も正確（Client Component で PATCH 呼び出し・router.refresh）

✅ **L01 通過により実装は実証済み**：
- 法律作成機能は動作確認済み
- Server Component の認証・DB 接続も正常

#### テスト前準備の必須タスク
オーディット実施前に以下を実行すること：
```sql
-- テストアカウント間のフレンド関係を作成
INSERT INTO public.friend_requests (sender_id, receiver_id, status)
SELECT
  (SELECT id FROM profiles WHERE email = 'e2e_user_a@example.com'),
  (SELECT id FROM profiles WHERE email = 'e2e_user_b@example.com'),
  'accepted';

-- 相互方向も推奨
INSERT INTO public.friend_requests (sender_id, receiver_id, status)
SELECT
  (SELECT id FROM profiles WHERE email = 'e2e_user_b@example.com'),
  (SELECT id FROM profiles WHERE email = 'e2e_user_a@example.com'),
  'accepted';
```

#### 実装ではなく環境の問題
- L02/L03/L04 の失敗は、招待が発生していないため
- コードレベルでは正しく実装されている
- テスト環境を整備すれば全て通過するはず
