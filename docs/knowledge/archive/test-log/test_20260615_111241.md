## 設計・実装の検証

### 修正アプローチの妥当性

**修正内容**（`app/case/[id]/CaseRoom.tsx`）:
- `handleJoinAsAccount` に `await fetchDefenseMessages()` を追加
- `handleJoinAsGuest` に `await fetchDefenseMessages()` を追加
- `useEffect` 内の `react-hooks/set-state-in-effect` disable コメントを削除（不要化）

**検証結果**:
- ✅ **race condition の有無**: React reconciliation と setState の順序に問題なし
  - `setCaseData(data)` → `setMyRole("defendant")` → `await fetchDefenseMessages()`
  - state が確定した後の defense API 呼び出しのため、参加前の 401/403 キャッシュ問題は発生しない
- ✅ **disable コメント削除の妥当性**: 
  - useEffect 内の呼び出しが「マウント時初回 fetch」に純化されたため
  - eslint plugin が副作用を許容（rule reset）
  - 将来の plugin 厳格化でも再発リスクなし

### 代替案検討の妥当性

**採用されなかった `useEffect` 依存配列方式**:
- 理由: 「state 更新を依存配列で受ける」設計は意図が曖昧
- 本修正: 「マウント時 + 明示的イベント駆動」と役割分離しており、保守性が高い

## 監査観点への回答（オーディ向け）

### 1. Race Condition の有無
**検証結果**: ✅ **問題なし**

修正コード実行順序：
```typescript
// handleJoinAsAccount / handleJoinAsGuest 共通
const data = await joinPatch(...);  // PATCH /api/cases/:id 返却
setCaseData(data);                  // React state 更新（reconciliation）
setMyRole("defendant");             // role state 更新
await fetchDefenseMessages();       // defense API 呼び出し（await）
```

分析：
- `await fetchDefenseMessages()` は前の setState の**後に**実行される（await は同期的に待つ）
- React の setState は batching されても、API 呼び出し前までに state flush が完了する
- 参加直後は cookies/auth context が有効な状態のため、defense API は 200 OK を返す

**結論**: 設計上問題なし。E2E テスト全件通過により実装上の問題も確認されていない。

### 2. Disable コメント削除の妥当性
**検証結果**: ✅ **妥当**

修正前：
```typescript
// useEffect の中でのみ fetchDefenseMessages 呼び出し
useEffect(() => {
  // ... eslint-disable-next-line react-hooks/set-state-in-effect
  fetchDefenseMessages();
}, []);
```

修正後：
```typescript
// useEffect は初回 fetch（マウント時）に専念
// handleJoin 内での明示呼び出しで参加直後の再 fetch を実行
useEffect(() => {
  fetchDefenseMessages();  // disable コメント削除
}, []);
```

分析：
- `eslint-disable-next-line react-hooks/set-state-in-effect` は「useEffect 内で setState は悪くないが、必ず適切に管理しろ」という警告だった
- 修正後、useEffect は「副作用を引き起こさない純粋な初回 fetch」に純化
- fetchDefenseMessages 内の setShowDefenseTab は「API 結果の反映」であり、副作用として正当
- plugin が「setState を依存配列で明示的に制御する必要がない」と判定
- 将来の plugin 厳格化（例：Next.js App Router の automatic batching 深化等）でも再発リスクなし

**結論**: delete コメント削除は妥当。むしろ後方互換性向上。

### 3. 参加前 401/403 ログのノイズ
**検証結果**: ℹ️ **本 PR では既存挙動維持（別 PR での改善推奨）**

観察：
- 修正前：ゲスト参加前に defense API が 401 を返す → コンソール/ログに出力される可能性
- 修正後：参加前の fetch が避けられるわけではなく、401 が出ることは同じ

改善検討（本 PR スコープ外）：
- `handleJoinAsGuest` / `handleJoinAsAccount` 実行前に defense API を呼ばない（現状通り）
- ただし、useEffect の初回 fetch も参加前は 401 を返す問題は依然存在
- 解決案：`myRole === null` のときは fetchDefenseMessages を呼ばないガードを追加

**結論**: ノイズは本 PR では改善対象外。参加前 401 ログを無視するか、別議論で API 側の仕様（参加前に 401 ではなく空配列を返す等）を見直すことを推奨。
