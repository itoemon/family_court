## 詳細分析

### Middleware による認証保護の問題

**ファイル**: `middleware.ts:32-38`

```typescript
const PROTECTED_PATH_PREFIXES = ["/history", "/profile", "/case"];
const isProtected =
  pathname === "/" ||
  PROTECTED_PATH_PREFIXES.some(p => pathname === p || pathname.startsWith(p + "/"));
if (!user && isProtected) {
  return NextResponse.redirect(new URL("/auth/login", request.url));
}
```

**問題**: `/case/*` パスがログイン必須（Protected）として扱われており、認証なしのアクセスを即座にログインページにリダイレクト

**設計との矛盾**:
- **要件** (requirements.md §4): 被告はゲストとして参加可能（名前入力のみ）
- **設計** (design.md): ゲスト参加フロー明記。GET /api/cases/:id で callerRole を判定し、未認証は "observer" 扱い。ケースページ（page.tsx:L255）は `!myRole && phase==="waiting"` のとき参加画面を表示
- **実装**: middleware が未認証アクセスを事前ブロック → ケースページにたどり着かない

**要件対応状況**:
- ✅ 原告のケース作成: 認証ユーザーのみで OK
- ✅ 被告のアカウント参加: 認証ユーザーのみ → middleware を通す
- ❌ 被告のゲスト参加: **未認証状態で URL アクセスが必要だが、middleware がブロック**

## オーディへの注意点

middleware の修正が必要。`/case/*` パスを以下のいずれかに変更:

**案 A**（推奨）: `/case` から保護を外す
```typescript
const PROTECTED_PATH_PREFIXES = ["/history", "/profile"];  // /case を削除
```
理由: ケースページ自体は未認証アクセスをサポート（callerRole 判定・observer 扱い）。中身（発言フォーム等）で権限チェック済み。

**案 B**: `/case` 配下の特定パスのみ保護
```typescript
const PROTECTED_PATH_PREFIXES = ["/history", "/profile", "/case/new"];  // 作成画面のみ保護
```
理由: ケース作成（POST /api/cases）は認証必須だが、既存ケース閲覧はゲスト許可

ビルド側で確認が必要。middleware 修正がスコープ外の場合、設計/要件の矛盾を指摘する。

## 総評

**CRITICAL シナリオ: 4件中2件通過 → パイプライン差し戻し対象**

nonce ベースゲストトークンの実装自体は設計通り（`lib/guest-token.ts`・API Routes の async/await 対応は正常）だが、ゲスト参加フロー全体が middleware によってブロックされており、CRITICAL-M03・M04（ゲスト被告フロー）が機能しない。

**差し戻し理由**:
- 要件「ゲストとして参加できる」の実装が不完全
- middleware による認証リダイレクトがゲスト参加フローをブロック
- M03・M04 は本来ゲスト参加機能の検証だが、参加に至らない

**期待される修正**:
1. middleware.ts で `/case` パスの保護を外すか緩和
2. または、ゲスト参加フロー全体の設計変更（ただし要件変更となる）

ビルド → アーキ → リードの連携で、要件と実装のギャップを埋めること。
