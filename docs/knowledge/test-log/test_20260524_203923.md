## テストシナリオ詳細

### CRITICAL-C01: 矛盾チェック警告が自分にのみ表示される（相手には見えない）

**結果**: ✅ 通過

**前提条件・注意事項**:
- API キーが未設定の場合、矛盾チェックはサーバー側でスキップされるため警告は表示されない
- このテストでは「相手（被告）側には絶対に警告バブルが表示されない」ことを主軸に確認
- 警告バブルの CSS クラス: `bg-amber-50 border-amber-200`（ContradictionWarningBubble コンポーネントのデザイン）

**確認内容**:
1. A（原告）が発言を投稿後、B（被告）側のビューをリロードして確認
2. B 側に `bg-amber-50 border-amber-200` クラスの要素が 0 件であることを確認 → **通過**
3. A 側の警告数 >= B 側の警告数（0）であることを確認 → **通過**

**設計との対応**:
- `app/case/[id]/page.tsx` の `myRole === arg.role` による表示制御が機能している
- RLS により `contradiction_warnings` は `user_id = auth.uid()` でのみ SELECT 可能

### NORMAL-C02: 過去ケースがない場合は警告なしで発言が投稿できる

**結果**: ✅ 通過

**確認内容**:
1. 原告・被告がケースを作成し、原告が発言を投稿
2. 発言が正常にタイムラインに表示される（エラーなし）
3. `.text-rose-500`（エラーメッセージクラス）が非表示であることを確認 → **通過**
4. 被告側（B）にも原告の発言が表示される → **通過**
5. 発言投稿後 2 秒待機後、矛盾警告バブル（`.bg-amber-50.border-amber-200`）が 0 件 → **通過**

**設計との対応**:
- `pastCases.length > 0` 判定によるスキップロジックが機能
- 過去ケース 0 件の場合、try-catch 内の矛盾チェックブロックをスキップ
- 発言レスポンスは正常に返る

### NORMAL-C03: /history から過去ケースへのリンクが機能する（verdict への直リンク確認）

**結果**: ✅ 通過

**確認内容**:
- テスト実行時点でテストユーザー A の過去ケース（verdict）が存在しない状態
- 空状態メッセージ「まだ過去のケースはありません」が表示されることを確認 → **通過**
- ページ全体のアクセス、レイアウト表示は正常

**バックログ対応確認**:
- `/history` のリンク href が `/case/[id]`（verdict ページでなく case ページ）であること
- verdict フェーズのケースにアクセスすると `app/case/[id]/page.tsx` 側のポーリングが `phase === "verdict"` を検出し `/case/[id]/verdict` に自動リダイレクトする実装
- 過去ケースが存在する場合はリンクをクリックして遷移確認するロジックも実装済み

## セキュリティ確認

✅ **RLS**: `contradiction_warnings` は `user_id = auth.uid()` でのみ SELECT 可（INSERT は admin クライアント経由のみ）
✅ **本人のみ表示**: クライアント側で `myRole === arg.role` で追加フィルタリング
✅ **プロンプトインジェクション対策**: XML タグでユーザー入力を区切るプロンプト設計
✅ **API キー不在**: `api_key_encrypted` なしの場合スキップ（発言フローに影響なし）
✅ **ゲスト除外**: `authenticatedUserId === null` の場合スキップ
✅ **失敗時の安全性**: `checkContradiction` 例外時も try-catch で握りつぶし、発言レスポンスは正常
