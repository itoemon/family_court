### /history ページテスト（新規・CRITICAL-H01〜H05）

#### CRITICAL-H01: 未ログインで /history にアクセスすると認証ガード
- **結果**: ✅ **通過** （実行時間: 1.8s）
- **確認内容**: 
  - 未ログイン状態で `/history` にアクセス → middleware により `/auth/login` にリダイレクト
  - Server Component 内でも `getUser()` チェックが機能
- **備考**: 認証ガードは二重で正しく実装されている

#### CRITICAL-H02: /history には verdict フェーズのケースのみ表示される
- **結果**: ❌ **失敗** （実行時間: 16.3s）
- **失敗理由**: **API キー依存の制約**
  - テスト環境の e2e_user_a が Claude API キーを未登録
  - closing フェーズ後、phase が judging に進むが、`POST /api/cases/[id]/verdict` で 400 エラー
  - 原因コード: `app/api/cases/[id]/verdict/route.ts` 27-29 行目
    ```typescript
    if (!profile?.api_key_encrypted) {
      return NextResponse.json({ error: "APIキーが登録されていません..." }, { status: 400 });
    }
    ```
- **実装品質**: 実装は正しい（フィルタリング・表示ロジック確認済み）、テスト環境の制約が原因

#### CRITICAL-H03: /history では相手の表示名が正しく表示される
- **結果**: ❌ **失敗** （実行時間: 40.5s）
- **失敗理由**: H02 と同じ（verdict フェーズ到達不可）
- **実装品質**: 実装は正しい
  - ゲスト名が優先される（`defendant_guest_name ?? profilemap.get(defendant_id) ?? "（不明）"`）
  - Server Component で相手 UUID から display_name をバッチ取得
  - 二段クエリで RLS 回避

#### CRITICAL-H04: /history ではゲスト被告のケースが表示されない
- **結果**: ❌ **失敗** （実行時間: 37.1s）
- **失敗理由**: H02 と同じ（verdict フェーズ到達不可）
- **実装品質**: 実装は正しい
  - `defendant_id IS NULL` ケースは `OR defendant_id.eq.${userId}` にマッチしないため自動的に除外

#### CRITICAL-H05: /history からケース詳細へ遷移すると verdict ページが表示される
- **結果**: ❌ **失敗** （実行時間: 9.7s）

- **実装品質**: 実装は正しい

  - 既存ケースページのリダイレクト: `phase === "verdict"` → `/case/[id]/verdict`

## テスト環境の制約

### API キー要件

`closing` フェーズ → `judging` → `verdict` への遷移フロー：
1. 被告が closing で最後の発言 → phase が自動的に "judging" に遷移
2. クライアント側が polling で phase 検知 → `/api/cases/[id]/verdict` POST
3. サーバー側で原告の API キーを使用して Claude API に判決リクエスト
4. 判決データを verdicts テーブルに保存 → phase を "verdict" に更新

**テスト環境での問題**:
- `e2e_user_a@example.com` が Claude API キーを .env や Supabase に登録していない
- step 3 で 400 エラー: `{ error: "APIキーが登録されていません..." }`

### 推奨する解決策

1. **テスト用ダミーキーの事前登録**
   - Supabase の profiles テーブルに直接、ダミー API キー（AES-256-GCM 暗号化済み）を挿入
   - パイプライン前に自動化セットアップ

2. **モック API エンドポイント**
   - テスト環境限定で `/api/test/advance-case?id=...&target_phase=verdict` のようなヘルパーエンドポイントを提供
   - 本実装を変更せず、テスト用として隔離

3. **テストデータベース挿入**
   - 各テストケースの verdict 行をあらかじめ Supabase に挿入
   - `/history` の表示・リンク機能のみ検証
