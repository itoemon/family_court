## 実装上の判断・設計書からの逸脱

設計書（design.md）の仕様に完全準拠した。逸脱なし。

- `Ratelimit` / `Redis` インスタンスはモジュールスコープで初期化（Route Handler 関数外）
- `slidingWindow(30, "1 m")` で 1 分間 30 リクエスト制限
- `reset` はミリ秒 → 秒に `Math.ceil(reset / 1000)` で変換
- レートリミットチェックの挿入位置: 認証確認直後・クエリパラメータ検証前（設計書の処理フロー通り）
- 環境変数未設定時のフォールバックなし（`Redis.fromEnv()` が即エラーになる意図した挙動）

## テスタ・オーディへの注意点

### 事前確認（テスト前必須）

1. **環境変数の設定**: `UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN` が `.env.local`（ローカル）および Vercel 環境変数（本番）に設定されていること。未設定のまま起動すると `Redis.fromEnv()` がエラーをスローし、`/api/users/search` が 500 になる。
2. **Upstash Redis インスタンスの作成**: Upstash コンソールでデータベースを作成し、REST URL とトークンを取得する作業はダイチ側が実施する。

### 重点確認ポイント

1. **正常系（30 リクエスト以内）**: `GET /api/users/search?q=test` が 200 を返すこと。レスポンスに `X-RateLimit-*` ヘッダーが不要（付与しない設計）。
2. **レートリミット超過（429）**: 同一ユーザーで 31 回連続リクエストを送ると 429 が返ること。
3. **429 ヘッダー検証**: 429 レスポンスに以下のヘッダーが含まれること。
   - `X-RateLimit-Limit: 30`
   - `X-RateLimit-Remaining: 0`
   - `X-RateLimit-Reset`: Unix epoch 秒（ミリ秒ではないこと）
   - `Retry-After`: 正の整数（0 以上）
4. **ユーザー単位の分離**: ユーザー A が 30 リクエストを消費しても、ユーザー B はブロックされないこと（user.id 単位であることを確認）。
5. **slidingWindow の動作**: 窓境界（毎分0秒）のリセット直後に連続リクエストを送っても、直近1分間のカウントが正しく計算されること。
6. **既存の認証チェック**: 未認証（セッションなし）では 401 が返ること（レートリミット前に弾かれる）。
7. **既存のクエリ検証**: `q` パラメータ欠如・101文字以上で 400 が返ること（レートリミット通過後に検証される）。

### セキュリティ観点

- **識別子は `user.id`**: NAT / VPN / プロキシ環境でも無関係ユーザーに影響しないことを確認。
- **環境変数の公開範囲**: `UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN` に `NEXT_PUBLIC_` 接頭辞がついていないこと（サーバー専用）。クライアントバンドルに混入していないことを確認すること。
- **`analytics: false`**: Upstash の使用量分析用データ送信を無効化している。デフォルトは `false` だが明示的に指定している。

### 既存テストへの影響

`test/` 配下に `/api/users/search` を叩く E2E テストがある場合、Upstash Redis への接続が必要になる。テスト実行環境に環境変数が設定されていないとモジュール読み込み時（`Redis.fromEnv()`）でエラーになる。
