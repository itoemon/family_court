## 指摘一覧

### [MEDIUM-001] `verifyGuestToken` 呼び出しが try-catch で保護されていない（`app/api/cases/[id]/argument/route.ts:35`）

- **内容**:  
  `argument/route.ts` の POST ハンドラ（35 行目）で `verifyGuestToken(id, cookieToken)` を呼んでいるが、try-catch で囲んでいない。  
  `verifyGuestToken` は内部で `computeToken` を呼び、`GUEST_TOKEN_SECRET` 未設定時に `new Error("GUEST_TOKEN_SECRET is not set")` を投げる。  
  この例外は POST ハンドラ内でキャッチされないため、Next.js のフレームワーク層に伝播する。  

  同じ `verifyGuestToken` を呼ぶ GET ハンドラ（`route.ts:57–73`）は全体を try-catch で囲み、`{ error: "サーバー設定エラーが発生しました。管理者に連絡してください。" }` を 500 で返す設計が貫かれている。  
  `generateGuestToken` を呼ぶ PATCH ハンドラ（`route.ts:114–120`）も同様に try-catch 済みである。  
  POST argument ルートだけ例外処理が欠落しており、設計書（`docs/knowledge/design.md:91`）の「各 API Route の catch ブロックで捕捉し 500 を返す」という方針と矛盾する。

- **影響範囲**:  
  - `GUEST_TOKEN_SECRET` が設定されていない環境でゲスト被告がケースに発言しようとすると、Next.js がランタイム例外を捕まえて 500 を返す。ただしレスポンスボディは JSON でなくなり、クライアント側の `data.error` 参照が機能しなくなる。  
  - **開発環境**では Next.js がエラーメッセージ（`"GUEST_TOKEN_SECRET is not set"`）をレスポンスに含めることがあり、環境変数名がエンドユーザーに露出する可能性がある（設計書の隠蔽方針に違反）。  
  - 本番環境（Next.js production ビルド）ではフレームワークが汎用 500 を返すためエラーメッセージは露出しないが、JSON ではないレスポンスが返るためエラーハンドリングの一貫性が損なわれる。

- **修正案**:  
  `argument/route.ts` の `verifyGuestToken` 呼び出しを try-catch で囲み、GET/PATCH ハンドラと同じエラーレスポンスを返す。

  ```typescript
  // 変更前（35行目周辺）
  if (!callerRole && c.defendant_guest_name) {
    const cookieToken = req.cookies.get(`guest_defendant_${id}`)?.value;
    if (cookieToken && verifyGuestToken(id, cookieToken)) {
      callerRole = "defendant";
    }
  }

  // 変更後
  if (!callerRole && c.defendant_guest_name) {
    const cookieToken = req.cookies.get(`guest_defendant_${id}`)?.value;
    if (cookieToken) {
      try {
        if (verifyGuestToken(id, cookieToken)) {
          callerRole = "defendant";
        }
      } catch (err) {
        console.error("verifyGuestToken failed:", err);
        return NextResponse.json(
          { error: "サーバー設定エラーが発生しました。管理者に連絡してください。" },
          { status: 500 }
        );
      }
    }
  }
  ```
