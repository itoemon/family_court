## 指摘一覧

### [LOW-001] 空文字列の `display_name` を持つユーザーの表示が新規 4 コンポーネント間で不整合（app/me/_components/FriendsCard.tsx:71 vs app/me/_components/ProfileCard.tsx:61, app/me/_components/MeHeader.tsx:26）

- **内容**: 
  - `MeHeader.tsx:26` の `const label = displayName ?? "（名前未設定）"` と `ProfileCard.tsx:61` の `{displayName ?? "（名前未設定）"}` は **`??`（nullish 合体）** を使うため、`display_name` が空文字列 `""` のとき空表示になる。
  - 一方 `FriendsCard.tsx:71` の `{f.displayName || "（名前未設定）"}` は **`||`（OR）** を使うため、空文字列のときフォールバック文字列「（名前未設定）」を表示する。
  - 同一ユーザーが自分自身の `display_name = ""` だった場合、ページ上部の `MeHeader` と「プロフィール」カードでは名前欄が空白に見え、「フレンド」カード（自分が他人視点で映る経路はないが、他人の `display_name = ""` の場合に発火）では「（名前未設定）」と表示される。本ページ内で同じ概念（名前未設定）の見え方が分かれる。
  - `display_name` は UI 上必須項目で空文字列は稀想定だが、過去 PR の `profiles` テーブルには NOT NULL 制約はあっても空文字制約は無いため、`""` が DB に入り得る（FEAT-002 設計を確認した範囲では明示的な空文字弾きは無い）。

- **修正案**: 
  - 三箇所のフォールバック演算子を統一する。意味的には「null / undefined / 空文字列 すべてフォールバックに倒す」のが UX 妥当のため、`MeHeader.tsx:26` / `ProfileCard.tsx:61` を `||` ベースに揃える案が小コスト。
  - 例: `MeHeader.tsx:26` を `const label = displayName && displayName.length > 0 ? displayName : "（名前未設定）";` などに改修、もしくは `ProfileCard.tsx:61` を `{displayName || "（名前未設定）"}` に変更。
  - 本指摘は機微情報露出や認可崩れではなく内部一貫性の問題のため LOW。設計書「FEAT-005 対応」アクセシビリティ設計の「件数バッジ・役割バッジは可視テキストを内包する」方針との整合性も理由になる。
