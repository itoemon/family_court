## 指摘一覧

### [LOW-001] handleJoin の try/catch が fetchDefenseMessages の例外を「参加失敗」として表示する（app/case/[id]/CaseRoom.tsx:129-149, 156-174）
- **内容**: `handleJoinAsAccount` (L129-149) と `handleJoinAsGuest` (L156-174) の `try` ブロック末尾に `await fetchDefenseMessages()` (L144, L169) が追加された。`fetchDefenseMessages` は 401/403 を握り潰す実装（L212-215）だが、`fetch` 自体のネットワーク失敗や `res.json()` のパース失敗時には例外を投げる。この場合、すでに `setMyRole("defendant")` と `setCaseData(data)` が完了している（参加は実際には成功している）のに、`catch` (L145, L170) で `setError("エラーが発生しました")` が呼ばれ、ユーザーには「参加に失敗した」と見える。被告は再試行のため画面を再操作するが、すでに `myRole=defendant` のため参加画面ロジック（L279 の `!myRole && caseData.phase === "waiting"`）から外れ、メイン裁判室が表示される、という分かりにくい遷移を起こす。
- **修正案**: `await fetchDefenseMessages()` を独立した try/catch でラップし、defense fetch のエラーは UI エラー表示に出さず silent fail させる。例:
  ```ts
  setMyRole("defendant");
  setCaseData(data);
  try { await fetchDefenseMessages(); } catch { /* useEffect / polling で復旧する */ }
  ```
  もしくは setMyRole/setCaseData の前に呼び出すか、`finally` 後の `useEffect` 再走に任せる設計に変える。

### [LOW-002] eslint-disable コメント削除の根拠が同ファイル内 `fetchCase` useEffect と矛盾している（app/case/[id]/CaseRoom.tsx:91-98 vs L222-227）
- **内容**: 旧 useEffect (L222-227) の `// eslint-disable-next-line react-hooks/set-state-in-effect` を削除した根拠を handoff/eng-to-aud.md は「useEffect が初回 fetch 専用に純化されたため」と説明する。しかし `fetchCase` の useEffect (L91-98) は構造的に同じパターン（`async` な useCallback を await 後 setState する関数を useEffect から呼ぶ）でありながら disable コメントを保持している。`react-hooks/set-state-in-effect` ルールは呼び出し元の多さや「純化」を解析しないため、両者で挙動が異なる理由がない。`fetchDefenseMessages` 側だけ lint が黙る保証はなく、CI lint の version up 等で再発した場合 disable コメントを再追加する手戻りが発生する。
- **修正案**: 以下のいずれかで一貫性を持たせる。(a) ローカルで `npm run lint -- app/case/\[id\]/CaseRoom.tsx` を実行し、両 useEffect の lint 出力（PASS/FAIL）を handoff に明記する。(b) `fetchDefenseMessages` 側にも disable コメントを残しておき、`fetchCase` と挙動を揃える。(c) 逆に `fetchCase` 側の disable コメントも削れるなら削除し、本ファイルから react-hooks/set-state-in-effect の disable をゼロにする。

### [LOW-003] `handleJoinAsAccount` / `handleJoinAsGuest` がソース順で `fetchDefenseMessages` の宣言より前に出現する（app/case/[id]/CaseRoom.tsx:126, 152, 210）
- **内容**: `handleJoinAsAccount` (L126〜)、`handleJoinAsGuest` (L152〜) は `fetchDefenseMessages` (L210 で `useCallback` で宣言) を参照するが、ソース順では参照箇所 (L144, L169) の方が宣言より上に来ている。JavaScript の関数宣言ホイスティングとクロージャ解決のおかげで実行時には動作するが、`const fetchDefenseMessages = useCallback(...)` は `let`/`const` で TDZ（一時的デッドゾーン）が存在し、もし将来「`async function` の宣言を `const ... = async () => {...}` に書き換える」リファクタが入ると参照前アクセスで `ReferenceError` を起こす潜在リスクがある。また grep でコードを読む読者は「定義 → 参照」の順序を期待するため可読性も落ちる。
- **修正案**: `fetchDefenseMessages` の宣言（L210-220）を `handleJoinAsAccount` (L126) より上、`fetchCase` の useCallback (L81-89) の隣に移動する。動作変更なし。

## 総評
BUG-004 の修正は最小限の差分（+13/-3）で症状を解消する方針が妥当で、`await fetchDefenseMessages()` を `setMyRole` 直後に置く位置選択は state 確定タイミングと auth/guest cookie の発行タイミングの両方を満たしている。テスタが BUG-004-Account/Guest/Regression + CRITICAL-M01〜M04 を全 7 件 PASS させており、リグレッションリスクも実測で抑え込まれている。指摘 3 件はいずれも LOW（防御的コーディングと一貫性の話）で、HIGH/MEDIUM 該当なし。通過条件（HIGH 0 件かつ指摘 5 件以下）を満たすため **本 PR は監査通過**。
