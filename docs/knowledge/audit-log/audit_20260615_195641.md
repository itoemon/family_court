## 指摘一覧

### [LOW-001] design.md / eng-to-aud.md の `lastSpeakerRole` 解決方針記述が実装と乖離（docs/knowledge/design.md:2503-2509, docs/knowledge/handoff/eng-to-aud.md:39-43）
- **内容**:
  `docs/knowledge/design.md` の `## BUG-005 閉廷アナウンス条件の修正` セクション L2503-2509 では「`lastSpeakerRole` は `end-proposal` / `extension-vote` の呼び出し側で `arguments` テーブルを `is_greeting=false` で SELECT して取得し、`insertClosingJudgeMessage` の `params.lastSpeakerRole` に渡す」と明記されている。`docs/knowledge/handoff/eng-to-aud.md` L39-43（「### 2. `lastSpeakerRole` の解決」節）でも同方針を踏襲している。

  しかし実装は前回監査 LOW-001（`audit_20260615_192508.md`）の消化として、コミット `b7419e7` で `lastSpeakerRole` の `arguments` SELECT を呼び出し側から撤去し、`lib/case-closing.ts:6-9` の引数を `{ caseId, topic }` のみへ簡略化している。ヘルパー内部 (`lib/case-closing.ts:38-40`) では `generateJudgeMessage` 呼び出しのシグネチャ互換のため `plaintiffName: ""`、`defendantName: ""`、`lastSpeakerRole: "plaintiff"` のダミー値を埋めている。`lib/judge.ts:49-56` の closing プロンプトは `topic` のみを参照するため機能影響は無く、`grep -rn "trigger_type.*closing" app/ lib/` も `lib/case-closing.ts:56` の 1 箇所に収束しており実装本体に欠陥は無い。

  問題は **設計書（永続資料）の記述が現実装と一致していない** 点。次パイプラインで `## BUG-005` セクションを参照したアーキ / ビルドが「呼び出し側で `arguments` SELECT が必要」と誤読してリファクタリングを誤る、または `lib/judge.ts` の closing プロンプトを `lastSpeakerRole` 依存に拡張する際にヘルパー側の簡略化に気づかず矛盾を持ち込むリスクがある。

- **修正案**:
  本タスクで設計書を更新する場合（オーディ書き込み権限は audit-log のみのためリードに引き継ぐ形）は、`docs/knowledge/design.md` の `## BUG-005` セクションに「**実装での簡略化注記**: `lib/case-closing.ts` は `{ caseId, topic }` のみを受け取り、`lastSpeakerRole` の解決を呼び出し側から撤去した。closing プロンプト (`lib/judge.ts:49-56`) が `topic` 以外を参照しないことを前提とした最適化（前回監査 `audit_20260615_192508.md` LOW-001 消化）。今後 closing プロンプトを `plaintiffName` / `defendantName` / `lastSpeakerRole` 依存に拡張する場合は、本ヘルパーの引数と本注記の両方を改訂すること」を追記する。`docs/knowledge/handoff/eng-to-aud.md` L39-43 にも同等の注記を加える（ただし handoff は揮発資料なので次パイプラインで書き換わる前提なら必須ではない）。

## 総評
- 実装本体は引き続き堅牢。`lib/case-closing.ts` のテーブル境界保護（`arguments` テーブル / `DEFAULT_CLOSING_GREETING` / `cases` UPDATE / 認可判定への参照なし）、`app/api/cases/[id]/argument/route.ts:130-163` の `nextPhase === "argument"` ガードによる `extension_voting` 遷移時の `judge_messages` INSERT 完全排除、`app/api/cases/[id]/end-proposal/route.ts:108-127` および `app/api/cases/[id]/extension-vote/route.ts:150-163` での `phase=judging` 楽観ロック（`WHERE phase=argument` / `WHERE phase=extension_voting AND 両者票一致`）による AI 閉廷宣告 INSERT 経路の重複防止、両経路で `insertClosingGreetingsForCase` → `insertClosingJudgeMessage` の順序遵守、失敗時にも `phase=judging` を維持するエラーハンドリングが、いずれも task.md の確定事項と design.md の方針通りに実装されている。前回 LOW-001 を消化したリファクタの方向性も妥当で、`lib/case-closing.ts:38-40` のダミー値補完はコメントで「シグネチャ互換のため」と明示しており、意図が後追い可能。
- セキュリティ観点でも、平文 API キー (`plaintiffApiKey`) はヘルパー引数として一度渡るのみで関数内に保持されず、`console.error` / `console.warn` ログにユーザー入力 / API キー / PII を載せていない（プレフィックス `[judge] closing:` と `case=<id>` / 例外オブジェクトのみ）。`decryptApiKey` 呼び出しも `plaintiffProfile?.api_key_encrypted` の non-null チェック後にのみ実行されている。
- 通過条件（HIGH 0 件、指摘 5 件以下）を満たすため **通過判定**。前回 LOW-003（API キー SET 経路の E2E 動的検証が現状環境で実行されない）は `docs/backlog.md` に `LOW-001-BUG005` として登録済みで、本タスクの本質価値（`phase=judging` 直後の `judge_messages` INSERT と `arguments` → `judge_messages` の時系列順序）がリグレッション検知できていないギャップは残るが、`tests/e2e/bug005-closing-trigger.spec.ts` で条件分岐 hard assertion 自体は組まれており、テスト Supabase の `e2e_user_a` プロフィールに有効な `api_key_encrypted` をセットアップする運用整備が完了し次第カバーされる見込み。本パイプラインで再指摘はしない。
