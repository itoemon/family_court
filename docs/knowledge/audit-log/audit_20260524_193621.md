## 指摘一覧

### [LOW-001] Supabase エラーオブジェクト素投げ（サーバーログなし）（app/history/page.tsx:40）
- **内容**: `if (error) throw error;` により、Supabase クエリ失敗時に生のエラーオブジェクトをそのまま投げている。本番環境では Next.js のエラーバウンダリが処理するためエラー詳細はユーザーに露出しないが、DB 接続障害・権限エラー等が発生しても可観測性がゼロ。障害時に原因特定が困難になる。
- **修正案**:
  ```typescript
  if (error) {
    console.error("[history] cases query failed:", error);
    throw new Error("ケース一覧の取得に失敗しました");
  }
  ```
  汎用エラーを再スローすることで、エラー詳細はサーバーログに残しつつ、クライアントへの情報漏洩を防ぐ。

---

### [LOW-002] middleware の保護パス判定が完全一致のみ（middleware.ts:32-34）
- **内容**: `PROTECTED_PATHS.has(pathname)` による完全一致判定のため、将来 `/history/` （末尾スラッシュ）や `/history/[sub]` などのサブルートが追加された場合、middleware によるリダイレクト保護が適用されない。現時点ではサブルートが存在せず、Server Component 側の `if (!user) redirect(...)` が二重保護として機能しているため即時影響はない。ただし、将来の開発者が Server Component チェックを省いてサブルートを追加するリスクがある。
- **修正案**:
  ```typescript
  if (!user && (pathname === "/" || pathname.startsWith("/history"))) {
  ```
  または、将来のサブルート追加に備えて Set → プレフィックスマッチに変更し、他の認証必須パス（`/profile`、`/case/new`）も同様に保護対象へ移行することを検討する。

---

### [LOW-003] プロフィール取得クエリのエラーが無言で握りつぶされる（app/history/page.tsx:55-63）
- **内容**: 
  ```typescript
  const { data: profiles } = await admin
    .from("profiles")
    .select("id, display_name")
    .in("id", Array.from(opponentIds));
  ```
  デストラクチャリングでエラーを無視している。クエリ失敗時は `profileMap` が空になり、全ケースの相手名が「（不明）」と表示される。エンドユーザーには「相手の名前が取得できなかった」という誤情報が表示され続けるが、エラーはログに残らない。
- **修正案**:
  ```typescript
  const { data: profiles, error: profileError } = await admin
    .from("profiles")
    .select("id, display_name")
    .in("id", Array.from(opponentIds));
  if (profileError) {
    console.error("[history] profiles query failed:", profileError);
  }
  ```
  処理は継続（「不明」表示で縮退）させつつ、ログには残す。

## 総評

HIGH・MEDIUM ゼロ。認証・認可・UUID 露出防止の3点は設計通りに実装されている。

- **認証二重ガード**: middleware の `PROTECTED_PATHS` チェックと Server Component の `getUser()` チェックが連動しており、未ログインアクセスは確実にブロックされる
- **認可（データ漏洩防止）**: `createAdminClient()` 採用の理由（profiles テーブル RLS による cross-user 読み取り不可）はビルド引き継ぎに明記されており妥当。WHERE 句 `plaintiff_id.eq.${user.id} OR defendant_id.eq.${user.id}` で自分のケースのみに限定されており、アプリ層フィルタが正しく機能している
- **UUID 露出防止**: `HistoryCase` 型に `plaintiff_id`/`defendant_id` が含まれておらず、クライアントへの不要な UUID 露出を防いでいる

LOW 3件はすべてサイレント障害時の可観測性に関するもので、現時点でエンドユーザーへのセキュリティ影響はない。
