## 指摘一覧

### [LOW-001] `handleJoinAsAccount` の未認証 early-return パスで `setLoading(false)` が呼ばれずローディング状態が残る（app/case/[id]/CaseRoom.tsx:150-151）
- **内容**: 旧実装は `finally { setLoading(false); }` で全ての退出経路に対して loading を解除していたが、本 PR で finally を廃止し catch 内 + try ブロック後の二箇所で setLoading(false) する構造に変えたため、`if (!user) { router.push(`/auth/login?next=...`); return; }`（L150-151）の経路では setLoading(false) が呼ばれない。catch にも入らず、try ブロック後の setLoading(false) も `return` でスキップされる。`router.push` での画面遷移までの間「参加中...」表記がボタンに残ったまま戻る／同期されたままになる可能性がある。同様の `if (!user) ... return;` パスは `handleJoinAsGuest` には存在しないため当該関数のみの問題。
- **修正案**: 該当行を `if (!user) { setLoading(false); router.push(`/auth/login?next=/case/${caseId}`); return; }` に変更する。router.push 直後にコンポーネントが unmount する想定でも、navigation 完了までのフリッカ防止として明示的に解除しておく方が finally 廃止前の挙動と一致する。

### [LOW-002] 「silent fail で polling/再マウントで復旧」コメントの「polling」が実態と一致しない（app/case/[id]/CaseRoom.tsx:170, 197）
- **内容**: `handleJoinAsAccount` (L170 周辺) と `handleJoinAsGuest` (L197 周辺) の `try { await fetchDefenseMessages(); } catch { /* silent: 後段で復旧 */ }` のコメントおよび handoff/eng-to-aud.md の LOW-001 対応説明では「polling/再マウントで復旧する」と書かれている。しかし `fetchDefenseMessages` を呼ぶ useEffect (L112-118) の依存配列は `[fetchDefenseMessages]` のみで、`fetchDefenseMessages` は `useCallback([caseId])` のため caseId が変わらない限り再生成されない。つまり**マウント時 1 回しか走らない設計で、polling は存在しない**（polling されているのは `fetchCase` のみ）。defense fetch がネットワーク失敗や JSON パース失敗で silent fail した場合、復旧経路はリロードしかなく、コメントの主張通りには動かない。実害は薄い（401/403 は別経路で握り潰されている）が、将来のメンテナでミスリードする。
- **修正案**: 以下のいずれか。(a) コメントを実態に合わせて「silent: 再マウント/リロードで復旧」に修正する（最も低コスト）。(b) 真に polling で復旧させたいなら `fetchCase` の `setInterval` 内で `showDefenseTab=false` の場合に `fetchDefenseMessages` も呼ぶ等の自己治癒経路を追加する。handoff/eng-to-aud.md の同記述も同時に修正することを推奨。

## 総評
前回オーディ (audit_20260615_111731.md) の LOW 3 件は、(1) `try/catch` 分離による defense fetch エラーの silent fail 化、(2) `fetchDefenseMessages` 側にも `eslint-disable-next-line react-hooks/set-state-in-effect` を残して `fetchCase` 側と整合、(3) `fetchDefenseMessages` の `useCallback` 宣言を `fetchCase` の隣（L91-101）へ移動して参照順を逆転解消、と全件適切に解消されている（CaseRoom.tsx の現行ソースで全て実物確認済）。テスタの BUG-004-Account/Guest/Regression + CRITICAL-M01〜M04 全 7 件 PASS の結果も維持されている。新規に検出した指摘は LOW 2 件で、いずれも「動作の正しさ」ではなく「コメントの正確性」「状態管理の網羅性」レベル。HIGH/MEDIUM 該当なし、指摘総数 2 件は通過条件（HIGH 0 件かつ指摘 5 件以下）を満たすため **本 PR は監査通過**。
