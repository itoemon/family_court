## 指摘一覧

### [HIGH-001] 矛盾警告メッセージが DB 保存前にトランケートされない（lib/contradiction.ts:38、supabase/migrations/.../create_contradiction_warnings.sql:6）

- **内容**: `checkContradiction` はモデルの出力テキストを `trim()` して返すだけで、50文字制限の強制が一切ない（lib/contradiction.ts:38）。プロンプトで「50文字以内」と指示しているだけであり、モデルが指示を無視した場合や、異なるモデルに差し替えた場合に制限を超えた文字列が `contradiction_warnings.message` に挿入される。DB側にも `CHECK (char_length(message) <= 50)` のような制約がない（migration:6）。エンドユーザー視点では、警告バブルが想定外の長文になりレイアウトが崩れる。また、`max-w-sm` の制約内に収まらない長文がユーザーに表示されるセキュリティ上の情報漏洩リスク（他ユーザーの過去発言内容が要約されず警告文に混入するケース）もある。
- **修正案**:
  1. `lib/contradiction.ts` の return 直前でトランケートを追加する。
     ```ts
     return text === "なし" ? null : text.slice(0, 50);
     ```
  2. あわせてマイグレーションに DB 制約を追加する。
     ```sql
     CHECK (char_length(message) <= 50)
     ```

### [MEDIUM-001] API キー取得クエリが重複実行されている（app/api/cases/[id]/argument/route.ts:107〜138, 144〜150）

- **内容**: `profiles` テーブルへの `.eq("id", c.plaintiff_id).single()` クエリが同一リクエスト内で2回実行されている。1回目は judge メッセージ生成用（行107〜111、`display_name, api_key_encrypted` を取得）、2回目は矛盾チェック用（行144〜148、`api_key_encrypted` のみ取得）。2回目のクエリは不要であり、1回目の結果を再利用すれば DB ラウンドトリップを削減できる。本番環境での発言投稿レイテンシに直接影響する。
- **修正案**: 1回目クエリの結果 `plaintiffProfile` をスコープの外（try-catch 外）に引き出すか、矛盾チェックブロックに変数を受け渡すリファクタリングを行う。judge ブロックで取得した `plaintiffProfile` と `apiKey` を矛盾チェックブロックでも参照できるよう変数スコープを広げる。

### [MEDIUM-002] `buildCaseResponse` が admin クライアントで `userId` のみでフィルタリングしており、case_id との組み合わせ検証で十分だが、取得件数に上限がない（lib/case-response.ts:50〜55）

- **内容**: `contradiction_warnings` のクエリは `.eq("case_id", caseId).eq("user_id", userId)` でフィルタリングされているが、`.limit()` が指定されていない（lib/case-response.ts:50〜55）。1ケースで矛盾警告が大量に蓄積するシナリオ（例: 攻撃的な発言を大量に投稿し続けた場合）で、レスポンスペイロードが無制限に膨張する。ケースの `max_rounds` は最大でも数ラウンドだが、将来的な上限緩和を見越すと防御的なリミットが望ましい。
- **修正案**: クエリに `.limit(100)` を追加する（ケースの発言上限を考慮した合理的な値）。
  ```ts
  .order("created_at")
  .limit(100);
  ```

### [LOW-001] `callerRole` が `null` のまま矛盾チェックに渡される型リスク（app/api/cases/[id]/argument/route.ts:164）

- **内容**: `.eq("role", callerRole)` の `callerRole` は型が `Role | null` のままである（eng-to-aud.md にも既知事項として記載）。実行時には行50の `if (!callerRole) return 403` により到達不可だが、TypeScript の型チェックは通過しており、将来的にコードを移動・再利用した際に null が渡るバグの温床になる。
- **修正案**: 矛盾チェックブロック内で型アサーションまたは非 null アサーションを明示する。
  ```ts
  .eq("role", callerRole as Role)
  // または
  .eq("role", callerRole!)
  ```

## 総評

基本的なセキュリティ設計（RLS・admin クライアント経由の INSERT・プロンプトインジェクション対策の XML タグ分離・ゲスト除外）は適切に実装されており、設計書の意図を正しく反映している。

ただし HIGH-001 は **実害を伴う実装上の欠陥**であり、通過判定の条件を満たさない。モデルの出力を無検証で DB に保存・UI に表示することは、想定外のレイアウト崩壊とコンテンツ漏洩リスクを生む。`text.slice(0, 50)` の1行追加で解消できるため、早急な修正を要する。
