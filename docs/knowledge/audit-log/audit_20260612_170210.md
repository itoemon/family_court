# 監査レポート

## サマリー
- 指摘件数: HIGH 0件 / MEDIUM 3件 / LOW 2件

## 指摘一覧

### [MEDIUM-001] 終了提案「合意」経路の挨拶 INSERT と phase UPDATE が原子的でなく、リトライで重複挨拶が混入する（app/api/cases/[id]/end-proposal/route.ts:104-115）
- **内容**: 「相手が提案中 → 同意」分岐で `insertClosingGreetings(admin, c)`（104行目）を先に実行し、その後で `cases.phase = "judging"` の UPDATE（105-112行目）を行っている。両者は別クエリで、トランザクション境界がない。UPDATE が失敗（DB 接続障害・制約違反など）すると 500 を返すが `arguments` テーブルには既に挨拶 2 行（plaintiff/defendant の closing 挨拶）が INSERT 済みで残る。クライアントが再試行すると、SELECT で `c.phase === "argument"` かつ `end_proposed_by = 相手ロール` のままなので再び合意経路に入り、挨拶 2 行がもう一度 INSERT され、合計 4 行（plaintiff×2、defendant×2）になる。CaseRoom.tsx:548-565 の吹き出し描画はすべての `is_greeting=true` 行を描画するため、UI に重複挨拶が表示される。verdict prompt（lib/claude.ts:21-26）に渡される `arguments` にも重複が混入し AI 出力品質を劣化させる。
- **修正案**: (a) `cases.phase` の UPDATE を先行させ、`.eq("end_proposed_by", 相手のロール).eq("phase", "argument")` で楽観ロックを付け、`{ data, error }` の `data` 行数が 1 のときだけ挨拶 INSERT を実行する（楽観ロック失敗時は既に他経路で遷移済みなので何もしない）。(b) もしくは `arguments` への INSERT 時に「同一 case_id × phase='closing' × is_greeting=true」の重複を弾く制約（部分 UNIQUE INDEX）を追加し、二重 INSERT を DB 側で防ぐ。最低限の対応として (a) の楽観ロック追加を推奨。

### [MEDIUM-002] 延長投票の確定 UPDATE がエラーチェックされず、片側だけ挨拶 INSERT された不整合状態でレスポンス 200 が返る（app/api/cases/[id]/extension-vote/route.ts:117-145）
- **内容**: 両者投票が揃った後の分岐で、continue 経路（119-131行目）の `max_rounds += 3` と argument 復帰 UPDATE、ならびに finish 経路（135-144行目）の `insertClosingGreetings` 後の phase=judging UPDATE のいずれも、戻り値の `error` を捕捉していない（118・119・134・135 行目は `await admin.from(...).update(...).eq(...)` のみで `const { error } = ...` 形式になっていない）。UPDATE が失敗した場合でも処理は続行され、最終的に `buildCaseResponse` が返り 200 OK を返す。結果として `cases` は `phase = "extension_voting"`・両票確定のまま固定され、以後 87-88 行目の `if (myCurrentVote !== null) → 409` でクライアントから再投票できず、UI 復帰経路がなくなる。finish 経路では既に終了挨拶 2 行が INSERT 済みのため、後から DB を手作業で `phase = "judging"` に戻すと挨拶が二重になる懸念もある（MEDIUM-001 と同根）。
- **修正案**: 両 UPDATE を `const { error: updateErr } = await admin...` で受け、`updateErr` があれば finish 経路では挨拶ロールバック手段がないので 500 + ログ出力で打ち切る（あるいは MEDIUM-001 と同様に挨拶 INSERT より先に UPDATE を実行し、UPDATE 成功時のみ挨拶 INSERT する順序に組み替える）。continue 経路は副作用が UPDATE のみなのでエラー時 500 で打ち切れば足りる。

### [MEDIUM-003] 「合意」経路の phase=judging UPDATE に楽観ロックがなく、相手の撤回操作と競合すると撤回が無視される（app/api/cases/[id]/end-proposal/route.ts:105-112）
- **内容**: 自分が提案撤回（90-100行目）は `.eq("end_proposed_by", actor)` で楽観ロックされており、合意経路への取り違いを防いでいる。しかし合意経路の UPDATE（105-112行目）は `.eq("id", id)` のみで、`end_proposed_by` の現在値を WHERE 条件にしていない。SELECT（47-51行目）から UPDATE までの間に提案者が別タブで撤回した場合、自分側のリクエストは「相手が提案中である」前提のまま `phase = "judging"` を強制する。結果として相手の意思（撤回 = 続行）に反して話し合いが終了し、判決生成（CaseRoom.tsx:101-118 の polling 経由）まで進む。FEAT-006 仕様の根幹である「両者合意での早期終了」原則を破る。
- **修正案**: 合意経路の UPDATE に `.eq("end_proposed_by", 相手ロール).eq("phase", "argument")` を付与し、楽観ロック失敗時（更新 0 件）は「相手が提案を取り下げたため終了できません」の 409 を返す。同時に挨拶 INSERT は UPDATE 成功確認後にのみ実行する（MEDIUM-001 の修正と統合可能）。

### [LOW-001] 開始挨拶 INSERT がケース参加 UPDATE と原子的でなく、失敗時に挨拶なし状態で固定化される（app/api/cases/[id]/route.ts:144-145, 193-194）
- **内容**: 被告参加処理（PATCH）で `cases` を `phase = "opening"` に更新（144行目認証経路 / 193行目ゲスト経路）した直後に `insertOpeningGreetings` を呼ぶが、両者の間にトランザクション境界がない。`insertOpeningGreetings` 内の `admin.from("arguments").insert([...])`（route.ts の被参照 `insertOpeningGreetings` 38-55行目）が Supabase エラーを返した場合でも例外は出ず、戻り値の `error` を確認していないため処理は続行する。結果としてケースは `phase = "opening"` で固定され、画面上に開始挨拶 2 行が存在しないまま会話開始となる。119 行目の `if (c.defendant_id || c.defendant_guest_name) return 409` により被告は再参加できず、挨拶を後追い投入する経路もないため、設計の「opening 開始時に両者挨拶を表示」が満たされなくなる。
- **修正案**: `insertOpeningGreetings` 内の INSERT を `const { error } = await admin.from("arguments").insert([...])` で受け、エラー時はログ出力 + ステータスを返り値で伝える。呼び出し側でエラーなら `cases.phase` を `waiting` に戻して 500 を返すか、参加 UPDATE と挨拶 INSERT の順序を逆（挨拶 INSERT → 成功時のみ phase=opening UPDATE）にする。

### [LOW-002] 挨拶 INSERT 経路が end-proposal と extension-vote で重複定義されており、整合性ドリフトのリスクがある（app/api/cases/[id]/end-proposal/route.ts:128-164 と app/api/cases/[id]/extension-vote/route.ts:160-196）
- **内容**: `insertClosingGreetings(admin, c)` ヘルパー関数の本体（`profiles.closing_greeting` 取得・`resolveClosingGreeting`・`arguments` への 2 行 INSERT、`role`・`phase = "closing"`・`round = 0`・`is_greeting = true`）が、両ファイルで一字一句同じ実装として複製されている。将来「挨拶記録から AI 入力除外フラグを足す」「`round` を NULL に統一する」など仕様変更が必要になったとき、片方だけ修正する取りこぼしが発生しやすい。実際の脆弱性ではないが、MEDIUM-001/002 の修正と組み合わせて両所を変更する必要があるため、共通化しないと連動修正漏れの可能性が高まる。
- **修正案**: `lib/greetings.ts` に `insertClosingGreetingsForCase(admin, { caseId, plaintiffId, defendantId })` のような共通関数を切り出し、両 route から `import` で呼び出す。lib/greetings.ts には既に `resolveClosingGreeting` / `DEFAULT_CLOSING_GREETING` が集約されているため、配置面の親和性も高い。

## 総評

FEAT-006 の根幹要件（max_rounds 固定 3 回採番・終了提案トグル・延長投票・固定挨拶 4 経路）は、設計書（design.md FEAT-006 セクション）および arch-to-eng.md の指示に忠実に実装されている。`end_proposed_by` の `text + check` 採用、案 A（cases 2 カラム延長投票）、案 1（arguments.is_greeting）、案 A（1 migration 集約）の各設計判断もすべて整合し、type 定義（lib/types.ts:21-22, 59-61, 110-111）、snake→camel マップ（lib/case-response.ts:72-74, 85）、PHASE_LABELS 追加（CaseRoom.tsx:13-21）、ゲスト被告対応（end-proposal/route.ts:28-33, extension-vote/route.ts:28-33）も漏れなく実装されている。schema.sql（supabase/schema.sql:14-17, 67, 72-79, 109）への反映も migration と一貫している。

一方で、HTTP 経由で起こる「合意」「延長確定」処理は INSERT/UPDATE を逐次実行する設計のため、片方が成功してもう片方が失敗したときの不整合に弱い。MEDIUM-001〜003 はいずれも同じ構造課題（複数書き込みを楽観ロック・エラー検査なしで連結している）に起因しており、共通の修正パターン（UPDATE 先行 + 楽観ロック + 失敗時 INSERT スキップ）で 3 件まとめて潰せる。FEAT-006 はラウンド進行の臨界点（終了提案合意・延長確定）に新規 UPDATE 経路を 2 本追加しているため、ここを堅牢化することは仕様の中核である「両者合意で正しく遷移する」性質を守るために重要である。

通過条件（HIGH 0 件、指摘総数 5 件以下）を満たすため、本監査は **通過判定** とする。MEDIUM-001/002/003 を次回小規模パッチで解消することを推奨する。
