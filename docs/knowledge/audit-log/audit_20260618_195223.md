## 指摘一覧

### [MEDIUM-001] SECURITY DEFINER 判定関数が PostgREST RPC として認証ユーザーに露出し、メンバー/オーナー関係の boolean オラクルになる（supabase/migrations/20260618190000_feat004_fix_laws_rls_recursion.sql:19-46）

- **内容**:
  RLS 無限再帰の修正として `public.is_law_member(uuid, uuid)` / `public.is_law_owner(uuid, uuid)` を `SECURITY DEFINER`（RLS バイパス）で新設し、`GRANT EXECUTE ... TO authenticated`（45-46 行）している。両関数は **`public` スキーマ**に置かれているため、Supabase の既定構成（PostgREST 公開スキーマ = `public`）では `POST {SUPABASE_URL}/rest/v1/rpc/is_law_member` として **任意の認証ユーザーが直接呼べる RPC エンドポイント**になる。
  関数は引数 `p_user_id` を呼び出し元本人に限定していないため、認証ユーザーは `is_law_member('<law_id>','<他人の user_id>')` を呼び、**第三者が特定の法律のメンバー/オーナーであるか否か**を boolean で取得できる。これは `law_members` の SELECT ポリシー（design.md「メンバーのみ閲覧可」、本タスクでも未緩和と明言）が保護しているはずの「メンバー構成は非メンバーに見せない」境界（task.md セキュリティ設計 149 行、design.md 2911 行「公開法律でもメンバー情報・招待・提案・投票は非メンバーに見せない」）を、SD 関数経由で部分的に迂回する。
  影響度は、`law_id` と `user_id`（いずれも UUID で列挙困難）を既知でなければ照会できないため限定的だが、漏洩するのが「RLS で保護対象の人間関係（誰がどの法律に属するか）」である点で MEDIUM とする。条文等のコンテンツ漏洩はない。

- **修正案**:
  RLS が参照する判定関数を PostgREST が公開しないスキーマへ退避する。例: `private` スキーマを作成し関数をそこへ移動（`private.is_law_member` 等）、`GRANT EXECUTE ON FUNCTION private.is_law_member(uuid,uuid) TO authenticated`、各ポリシーの参照を `private.*` に張り替える。RLS は引き続き機能し、PostgREST の RPC 公開からは外れる（`db-schemas` に `private` を含めない既定構成）。これが Supabase 公式が推奨する「RLS ヘルパーは非公開スキーマに置く」定石。スキーマ移動が困難なら、最低限 PostgREST 公開スキーマからの可視性を断つ運用（`REVOKE EXECUTE ... FROM anon` は既存だが、`authenticated` への RPC 公開自体を塞ぐ手当て）を別 migration で行うこと。

### [LOW-001] Hub 検索で検索語を初期値（空文字）に戻すと一覧が再取得されず、絞り込み結果が残留する（app/laws/hub/_components/HubSearch.tsx:21-23）

- **内容**:
  `useEffect` 冒頭の `if (query === initialQueryRef.current) return;`（22 行）により、検索語が初期クエリ（`/laws/hub` 直アクセス時は空文字）と一致した時点で `fetch` を行わず早期 return する。このため「`/laws/hub`（初期 `q=""`）→ "民法" と入力（フィルタ結果に差し替わる）→ 入力を全消しして空文字に戻す」操作で、`query === ""` が `initialQueryRef.current === ""` に一致して再取得が走らず、**検索ボックスは空なのに直前の絞り込み結果が表示され続ける**。ユーザーはリロードしない限り全件（新着 50 件）一覧に戻れない。エンドユーザーから見ると「検索をクリアしたのに一覧が元に戻らない」不具合として観測される。

- **修正案**:
  初回マウントのみ SSR 結果を再利用する意図は `initialQueryRef` で表現できているが、空文字復帰時は再取得すべき。`query` 空（正規化後 null 相当）のときは `initialLaws` に戻す分岐を入れるか、初回マウント判定を `useRef<boolean>(true)` のマウントフラグに変更し「マウント直後の 1 回だけスキップ、以降は値に関わらず debounce fetch」する方式へ改める。

### [LOW-002] 公開法律カードの日付描画が実行環境のタイムゾーン依存で、SSR/CSR 間のハイドレーション不一致を起こし得る（app/laws/hub/_components/PublicLawCard.tsx:16）

- **内容**:
  `new Date(law.created_at).toLocaleDateString("ja-JP")`（16 行）は実行ランタイムのローカルタイムゾーンで暦日を決める。`PublicLawCard` は `'use client'` の `HubSearch` 配下でレンダリングされるが、Next.js の Server Component による初期 HTML 生成時はサーバ（通常 UTC）で、ハイドレーション時はブラウザ（JST = UTC+9）で評価される。`created_at` が日付境界付近（UTC 15:00〜24:00 = JST 翌日 0:00〜9:00）の公開法律では、サーバ描画日とクライアント描画日がずれ、React のハイドレーション不一致警告および表示日付のちらつきが発生する。

- **修正案**:
  日付フォーマットを TZ 非依存にする。`created_at`（ISO8601）から `new Date(...).toISOString().slice(0,10)` で日付部分を切り出して整形するか、`Intl.DateTimeFormat("ja-JP", { timeZone: "Asia/Tokyo" })` のように `timeZone` を明示固定する。表示専用なので Server Component 側で整形済み文字列を props で渡す方式でも解消できる。

## 総評

FEAT-004 のセキュリティ設計の核（`laws_select_public` を `TO authenticated` 限定の PERMISSIVE ポリシーとして OR 追加し、既存 `laws_select_member_or_invitee` を書き換えない／`law_members`・`law_invitations`・`law_proposals`・`law_proposal_votes` の SELECT ポリシーを据え置く）は task.md・design.md の意図どおり実装されており、公開によって広がる範囲が `laws` 本体に限定されている点は確認した。Hub API のレスポンス境界型 `PublicLawListItem` が `owner_id` を持たず、`fetchPublicLaws` が整形時に `owner_id` を確実に破棄している点、import が元法律を読み取り専用参照のみで純クローン（`is_public=false` で INSERT）している点、`q` の LIKE 特殊文字エスケープ、`/laws/[id]` のメンバーシップ・ゲート（公開法律でも非メンバーは redirect）が未緩和である点も、いずれも仕様を満たす。実装中に顕在化した RLS 無限再帰を SECURITY DEFINER 関数化で恒久修正し、セマンティクスを保ったまま再帰連鎖を断った判断自体は妥当である。

一方で、その修正で導入した判定関数の **配置スキーマ**（`public`）が PostgREST 経由のメンバー関係オラクルという新たな露出面を生んでいる（MEDIUM-001）。RLS の再帰を関数で断つ場合は「関数を公開スキーマに置かない」までをワンセットで設計する必要がある。残る 2 件はいずれも Hub UI の挙動に閉じた LOW であり、HIGH 相当の認可破り・コンテンツ漏洩は検出されなかった。MEDIUM-001 は本リリースを止める性質ではないが、人間関係グラフの保護は本サブシステムの設計原則そのものであるため、近い改修サイクルでの是正を強く推奨する。
