## 指摘一覧

### [MEDIUM-001] ケースAPIが内部ユーザーIDを認証なしに公開（app/api/cases/[id]/route.ts:38、同:107、app/api/cases/[id]/argument/route.ts:107）

- **内容**:  
  `GET /api/cases/[id]` は認証不要なエンドポイントである（要件定義 §画面一覧: ケースルームは「認証: 任意」）。今回のコミットで `buildCaseResponse` に `defendantId: c.defendant_id ?? null` が明示的に追加され（route.ts:38）、さらに `select("*")` + `...c` スプレッドにより `plaintiff_id` / `defendant_id` を含む全DBカラムもそのままレスポンスに含まれる状態が継続している。  
  結果として、ケース IDを知る者であれば誰でも原告・被告の Supabase User UUID を取得できる。UUID は直接的な個人情報ではないが、複数ケースIDが判明しているケースでは同一 UUID の突き合わせにより同一ユーザーの参加状況（話し合い履歴の断片）を推測できる。  
  クライアント側の被告ロール復元ロジック（app/case/[id]/page.tsx:51-64）が `defendantId` を必要としている点は理解できるが、現在のユーザーが被告かどうかの判定はサーバー側エンドポイント（例: `/api/cases/[id]/my-role`）で完結させ、UUID 自体をクライアントに渡さない設計が望ましい。

- **修正案**:  
  1. `buildCaseResponse` の `select("*")` を必要カラムのみ列挙する `select` に変更し、`plaintiff_id` / `defendant_id` をレスポンスから除外する。  
  2. `defendantId` フィールドの代わりに `/api/cases/[id]/my-role`（GET、要認証）を新設し、現在のセッションユーザーが `plaintiff` / `defendant` / `none` のいずれかを返す設計に変更する。  
  3. page.tsx の `restoreRole()` はそのエンドポイントを叩くよう変更する。

### [LOW-001] ゲスト名（defendantName）に最大長バリデーションなし（app/api/cases/[id]/route.ts:87-90）

- **内容**:  
  `PATCH /api/cases/[id]` のゲスト参加パスで `body.defendantName` の最大長を検証していない。今回のコミットは当該コードブロックに Cookie 発行処理を追加しているが、長さ検証は追加されていない。同一コミット内で `topic`（200文字）と `content`（500文字）の上限検証が追加されており（app/api/cases/route.ts:14-16、app/api/cases/[id]/argument/route.ts:51-53）、ゲスト名だけが非一貫な状態にある。  
  PostgreSQL の `text` 型は最大 1 GB を格納可能であり、悪意ある入力により異常なサイズの文字列が `cases.defendant_guest_name` に書き込まれる可能性がある。ゲスト名は UI で表示されるため、極端に長い入力はレイアウト破壊を引き起こしうる。

- **修正案**:  
  `PATCH` ハンドラのゲスト参加パスに以下を追加する（他のバリデーションと同様のパターン）。  
  ```typescript
  if (body.defendantName.trim().length > 50) {
    return NextResponse.json({ error: "名前は50文字以内で入力してください" }, { status: 400 });
  }
  ```  
  フロントエンド側も `<input maxLength={50} />` を対応させること。

## 総評

HIGH-001 の修正は設計書の仕様（`getUser()` によるセッション検証、DB 照合によるロール導出、HMAC-SHA256 Cookie によるゲスト認証）を正確に実装しており、認可バイパスの問題は解消されている。`timingSafeEqual` の Buffer 長チェック、`SameSite=Strict` / `httpOnly` の Cookie 属性設定、`body.role` の完全廃止など、設計書の細部まで実装されている点は評価できる。

MEDIUM-001 は今回の変更で意図的に追加されたフィールドに起因する問題であり、次のビルドサイクルで対処が必要である。LOW-001 は今回触れたコード範囲の一貫性不足であり、合わせて修正することを推奨する。
